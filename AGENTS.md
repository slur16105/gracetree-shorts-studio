# GraceTree Shorts Studio Agent Guide

이 문서는 이 저장소에서 작업하는 모든 코딩 에이전트의 공통 행동 계약이다. 프로젝트 세부 계약은 아래 문서를 우선 확인한다.

- 제품 요구사항: `_bmad-output/planning-artifacts/prds/`
- 기술 결정과 구현 패턴: `_bmad-output/planning-artifacts/architecture.md`
- 승인된 Epic·Story와 인수 조건: `_bmad-output/planning-artifacts/epics.md`
- 현재 구현 계획과 승인된 변경: `_bmad-output/implementation-artifacts/`의 `status: ready-for-dev|in-progress|in-review|done`인 Spec, `status: approved-*`인 Sprint Change Proposal

디렉터리 안의 초안, 리뷰 프롬프트와 폐기 문서는 승인된 계약으로 간주하지 않는다. 필수 계약 문서가 없거나 최신·승인 상태를 판별할 수 없으면 구현을 중단하고 보고한다.

## 우선순위

충돌이 있으면 다음 순서로 따른다.

1. 현재 사용자의 명시적 지시와 승인
2. 승인된 Story, Spec, Sprint Change Proposal
3. Architecture와 PRD
4. 이 문서
5. 가까운 기존 코드 패턴

충돌이 결과나 범위를 바꾸면 임의로 선택하지 말고 보고한다.
아래의 프로젝트 안전 규칙과 금지는 최소 안전선이다. 상위 문서가 이를 완화하거나 상충하면 실행을 중단하고 명시적 승인을 받는다.

## 작업 전에

- 사용자 요청을 완료 조건과 금지 범위까지 읽는다.
- 관련 파일, 가까운 구현, 기존 테스트를 읽고 `git status --short`, staged·unstaged diff와 untracked 파일을 확인한다.
- 작업 시작 시 현재 HEAD와 기존 dirty 변경을 baseline으로 기록하고, 이번 작업이 추가한 범위와 구분한다.
- 기존 변경은 사용자 소유로 간주한다. 관련 없는 변경을 수정하거나 되돌리지 않는다.
- 사실, 코드에서 확인한 추론, 아직 검증되지 않은 가정을 구분한다.
- 불확실성이 제품 동작, 데이터 모델 또는 공개 계약을 바꾸면 구현 전에 질문한다.

## 구현 원칙

- 요청을 충족하는 가장 작은 완전한 변경을 만든다.
- 기존 구조, 명명, 경계와 유틸리티를 우선 재사용한다.
- 먼저 직접적인 구현을 선택하고 실제 중복이 확인된 뒤에만 추상화한다.
- 새 의존성, 설정 옵션과 범용 프레임워크는 필요성이 입증되고 승인 범위 안일 때만 추가한다.
- 국소 변경을 위해 전체 파일이나 문서를 재생성하지 않는다.
- 오류를 숨기는 fallback, 광범위한 catch, 무의미한 기본값으로 실패를 정상처럼 만들지 않는다.
- 임시 우회가 불가피하면 제한 범위, 제거 조건과 남은 위험을 명시한다.

## 프로젝트 안전 규칙

- renderer에 파일 시스템, SQLite 또는 프로세스 직접 접근을 추가하지 않는다.
- Electron main/preload/Python 경계에서 경로, 작업 ID와 명령 인자를 재검증한다.
- 원본 입력을 수정하거나 삭제하지 않는다. 기존 완료 결과는 승인된 재생성의 새 결과 검증과 원자적 커밋이 성공하기 전까지 수정하거나 삭제하지 않는다.
- 승인된 입력 교체·제거, 임시 파일 정리와 재생성은 앱 관리 루트 검증 및 Story의 안전 조건 안에서만 수행한다.
- 최종 경로에 직접 렌더링하지 않고 임시 결과를 검증한 뒤 커밋한다.
- DB 변경은 최초 필요 Story가 소유하는 순차 마이그레이션과 회귀 테스트로 구현한다.
- 프로토콜, 오류 코드, 상태 전이 또는 DB 스키마를 바꾸면 공유 계약과 테스트를 함께 갱신한다.
- 비밀값, 개인 데이터, 스크립트 전문과 입력 미디어 내용을 로그·문서·외부 서비스에 노출하지 않는다.

## 먼저 승인받을 변경

- 요구사항 또는 승인된 Story 범위를 확대하는 기능
- 아키텍처 결정, 공개 계약, 기술 스택 또는 DB 스키마 소유권 변경
- 새 런타임·프로덕션 의존성 추가
- 데이터 삭제, 기존 결과 교체, 파괴적 Git 명령 또는 외부 시스템 변경
- 실패 테스트 삭제·완화, 검사 비활성화 또는 품질 기준 하향

## 검증

- 변경된 동작에 가장 가까운 테스트부터 실행하고 필요하면 통합·E2E 검증으로 확장한다.
- DB 마이그레이션은 빈 DB와 직전 스키마 DB, 공유 계약은 양쪽 언어의 계약 테스트, 패키징은 대상 플랫폼 스모크 테스트를 필수 검증으로 취급한다.
- 버그 수정은 가능하면 실패를 재현하는 테스트를 먼저 추가한다.
- 테스트를 통과시키기 위해 제품 동작이나 테스트의 의미를 왜곡하지 않는다.
- 명령의 종료 코드와 핵심 출력을 직접 확인한다.
- 실행하지 않은 테스트, 읽지 않은 파일 또는 확인하지 않은 결과를 검증했다고 주장하지 않는다.
- 환경 문제로 검증하지 못한 항목은 원인과 영향을 완료 보고에 남긴다.

## 완료 기준

- 요청 범위와 인수 조건이 충족됐다.
- 관련 테스트, 타입 검사, 린트와 포맷 검사가 위험도에 맞게 통과했다.
- baseline 이후 이번 작업의 diff에 요청 밖 변경, 임시 디버그 코드, 비밀값과 의도하지 않은 생성 부산물이 없다.
- 오류 경로와 데이터 보존 조건을 확인했다.
- 완료 보고에 변경 파일, 실행한 검증 명령과 결과, 미검증 항목과 남은 위험을 간결하게 적는다.

## 금지

- 요청하지 않은 리팩터링, 추상화, 호환 계층, 기능 또는 의존성을 끼워 넣지 않는다.
- 증상만 사라지게 하는 무작위 변경을 반복하지 않는다.
- 실패를 숨기거나 부분 성공을 완료로 보고하지 않는다.
- 사용자 변경을 덮어쓰거나 승인 없이 파괴적 명령을 실행하지 않는다.
