# PRD: GraceTree Shorts Studio

## 0. 문서 목적

이 문서는 반복되는 말씀기도 YouTube Shorts 제작 작업을 자동화하는 개인용 PC 애플리케이션의 MVP 요구사항을 정의한다. 제품 기능과 검증 가능한 결과를 중심으로 기술하며, Next.js·Python·FFmpeg·Whisper 등 구현 후보와 기술적 쟁점은 별도 [addendum.md](./addendum.md)에서 후속 UX 및 아키텍처 입력으로 관리한다.

## 1. 비전

GraceTree Shorts Studio는 사용자가 준비한 이미지, 음성, 배경음악, 스크립트를 기존 영상 양식에 맞춰 조합하여 `1080x1920` 말씀기도 쇼츠 영상을 자동 생성하는 PC용 로컬 제작 도구다.

현재 CapCut에서 약 30분 동안 반복 수행하는 파일 배치, 영상 길이 조정, 자막 생성, 배경음악 합성 및 렌더링 작업을 단순한 파일 등록과 생성 실행으로 대체한다. MVP의 핵심 가치는 화려한 편집 기능이 아니라, 추가 편집 없이 게시 가능한 결과를 5분 이내의 사용자 작업으로 얻는 것이다.

초기에는 제작자 본인이 설치형 로컬 애플리케이션으로 실제 작업에 사용하며 자동화 품질과 반복 사용성을 검증한다. 충분한 효용과 상품화 가능성이 확인되면 다중 작업, YouTube 업로드, 다른 영상 양식 및 외부 사용자를 지원하도록 확장한다. MVP는 로컬 사용에 집중하되 영상 생성 기능이 특정 UI나 저장 방식에 과도하게 결합되지 않도록 하여 후속 서비스 전환 가능성을 보존한다.

## 2. 대상 사용자

### 2.1 주요 사용자

초기 사용자는 매일 동일한 양식의 말씀기도 쇼츠를 제작하는 제작자 본인 한 명이다. PC에서 영상 원본을 준비하고 결과 파일을 로컬에서 관리한다.

### 2.2 해야 할 일

- 반복적인 CapCut 편집 절차 없이 준비된 자료로 쇼츠 영상을 생성한다.
- 입력 자료가 충분하고 올바른지 생성 전에 빠르게 확인한다.
- 제작 중인 작업의 상태를 확인하고 완료 여부를 명확히 인지한다.
- 실패 시 원인과 해결 방법을 확인한 뒤 다시 시도한다.
- 완료된 영상의 로컬 폴더를 열어 재생, 이동 또는 삭제한다.
- 자동화 결과가 기존 제작 품질을 대체할 수 있는지 실제 사용으로 판단한다.

### 2.3 MVP 비대상 사용자

- 모바일에서 영상을 제작하는 사용자
- 여러 사용자와 프로젝트를 공유해야 하는 팀
- 서버 기반 SaaS나 원격 렌더링이 필요한 사용자
- 자유 형식 영상 편집기를 원하는 사용자

### 2.4 핵심 사용자 여정

**UJ-1. Slur가 게시 날짜에 맞춰 말씀기도 쇼츠를 생성한다.**  
Slur는 PC에서 애플리케이션을 열고 영상의 게시 날짜를 선택한 뒤 메인 화면의 큰 등록 영역에 해당 날짜의 입력 파일을 한 번에 드래그앤드롭한다. 시스템은 각 파일을 역할별로 분류하고 날짜별 작업 폴더에 보관한 뒤 준비 상태를 표시한다. 필수 자료가 모두 준비되면 Slur가 생성 버튼을 누른다. 화면은 현재 단계와 진행률을 표시하고, 완료되면 모달로 성공을 알린다. Slur는 완료 목록에서 게시 날짜와 결과 폴더를 확인한다.

**UJ-2. Slur가 누락되거나 잘못된 입력을 수정한다.**  
Slur가 파일을 등록했지만 필수 자료가 누락되거나 같은 역할의 파일이 중복되면 시스템은 생성 버튼을 활성화하지 않고 문제 항목과 해결 방법을 표시한다. Slur는 문제가 있는 파일을 제거하거나 다시 등록한 뒤 생성한다.

**UJ-3. Slur가 생성 실패를 복구한다.**  
생성 중 오류가 발생하면 시스템은 실패 단계, 이해 가능한 오류 원인 및 다음 동작을 모달로 안내한다. Slur는 입력을 수정하거나 환경 문제를 해결한 뒤 같은 작업을 다시 실행한다.

## 3. 용어

- **게시 날짜** — 쇼츠가 게시될 예정인 날짜. 작업을 구분하고 날짜별 작업 폴더와 완료 여부를 관리하는 기준이다.
- **입력 파일** — 한 개의 쇼츠 생성에 필요한 날짜별 자료. MVP에서는 썸네일 이미지, 음성 녹음 및 스크립트를 포함하며, 필요한 경우 작업별 배경음악을 추가할 수 있다.
- **공통 리소스** — 여러 쇼츠에 반복 사용하는 제목·말씀 영상, 기도 영상, 기본 배경음악 및 폰트.
- **작업** — 게시 날짜와 입력 파일 한 세트로 쇼츠 한 개를 생성하는 단위. MVP에서는 게시 날짜 하나당 완료 작업 하나만 허용한다.
- **입력 슬롯** — 등록된 파일이 배정되는 역할별 위치: 썸네일, 음성, 배경음악, 스크립트.
- **준비 상태** — 모든 필수 입력 파일과 공통 리소스가 유효하여 생성을 시작할 수 있는 상태.
- **생성 단계** — 입력 검증, 음성 분석, 자막 생성, 영상 합성, 결과 저장 등 작업 진행 구간.
- **완료 항목** — 성공적으로 결과 파일이 생성되어 완료 목록에 기록된 작업.
- **결과 폴더** — 최종 영상과 중간 산출물 및 로그가 저장되는 로컬 폴더.

## 4. 제품 기능

### 4.1 로컬 입력 등록 및 검증

**설명:** 사용자는 게시 날짜를 선택하고 메인 화면의 큰 등록 영역에 한 작업의 입력 파일을 함께 드래그앤드롭한다. 시스템은 파일을 입력 슬롯에 자동 배정하고 선택한 날짜의 작업 폴더에 복사해 보관하며, 준비 여부를 즉시 표시한다. 사용자가 6일치 자료를 미리 준비하더라도 MVP에서는 한 번에 한 쇼츠 작업만 등록하고 생성한다.

#### FR-1: 게시 날짜 선택

사용자는 입력 파일을 등록하고 영상을 생성할 게시 날짜를 선택할 수 있다. UJ-1을 실현한다.

**검증 조건:**

- 메인 화면에서 게시 날짜를 선택할 수 있다.
- 기본값은 현재 날짜이며 사용자가 다른 날짜로 변경할 수 있다.
- 선택된 게시 날짜를 입력 상태와 생성 화면에 계속 표시한다.
- 게시 날짜가 확정되지 않으면 생성할 수 없다.

#### FR-2: 날짜별 입력 파일 보관

시스템은 등록된 입력 파일을 선택한 게시 날짜의 작업 폴더에 복사하여 보관할 수 있다.

**검증 조건:**

- 게시 날짜별로 구분되는 작업 폴더를 생성한다.
- 등록된 파일을 해당 작업 폴더의 입력 영역에 복사한다.
- 애플리케이션을 다시 실행해도 해당 날짜에 등록된 입력 파일을 확인할 수 있다.
- 같은 날짜에 입력 파일을 다시 등록하면 기존 파일을 묵시적으로 덮어쓰지 않고 교체 여부를 확인한다.
- 사용자가 드래그앤드롭한 출처 파일은 변경하거나 삭제하지 않는다.

#### FR-3: 완료된 게시 날짜 중복 방지

시스템은 이미 완료된 게시 날짜에 새 영상을 중복 생성하지 않도록 한다.

**검증 조건:**

- 완료 작업이 있는 게시 날짜를 선택하면 이미 생성된 날짜임을 표시한다.
- 해당 날짜에는 새 생성 버튼을 비활성화한다.
- 기존 완료 항목과 결과 폴더를 열 수 있는 동작을 제공한다.
- 일반 생성 동작으로는 기존 결과를 묵시적으로 덮어쓰지 않는다.

#### FR-4: 완료 작업 다시 생성

사용자는 이미 완료된 게시 날짜의 입력 파일을 확인·교체한 뒤 명시적으로 영상을 다시 생성할 수 있다.

**검증 조건:**

- 완료된 게시 날짜를 선택하면 `다시 생성` 동작을 제공한다.
- 다시 생성 전에 기존 결과가 교체된다는 확인 모달을 표시한다.
- 사용자는 날짜별 보관 입력 파일을 그대로 사용하거나 일부 파일을 교체할 수 있다.
- 새 영상 생성이 성공하기 전까지 기존 완료 영상은 유지된다.
- 새 영상 생성이 성공하면 기존 결과를 교체하고 완료 기록의 실제 생성일을 갱신한다.
- 다시 생성에 실패하거나 취소되면 기존 완료 영상과 완료 기록을 유지한다.

#### FR-5: 다중 파일 드래그앤드롭 등록

사용자는 한 작업에 필요한 여러 입력 파일을 한 번의 드래그앤드롭으로 등록할 수 있다. UJ-1을 실현한다.

**검증 조건:**

- 드롭된 파일마다 등록 성공 또는 거부 상태가 표시된다.
- 파일 등록을 위해 각 입력 슬롯을 개별 클릭할 필요가 없다.
- 지원하지 않는 파일은 다른 입력 파일의 등록을 방해하지 않고 별도로 안내된다.

#### FR-6: 입력 슬롯 자동 분류

시스템은 등록된 파일을 썸네일, 음성, 작업별 배경음악, 스크립트 입력 슬롯으로 자동 분류할 수 있다. UJ-1과 UJ-2를 실현한다.

**검증 조건:**

- 이미지 파일은 썸네일 후보, 텍스트 파일은 스크립트 후보로 분류된다.
- 음성 녹음과 작업별 배경음악처럼 확장자가 겹칠 수 있는 파일은 기본 파일명 규칙 `voice.*`, `bgm.*`를 우선 적용한다.
- 역할을 확정할 수 없는 파일은 미분류 상태로 표시하고 해결 방법을 안내한다.
- 동일 입력 슬롯에 후보가 여러 개면 충돌 상태를 표시한다.
- 작업별 배경음악이 등록되지 않으면 설정된 기본 배경음악을 사용한다.

#### FR-7: 입력 준비 상태 검증

시스템은 생성 전에 모든 필수 입력 파일과 공통 리소스의 존재 및 기본 유효성을 검증할 수 있다. UJ-2를 실현한다.

**검증 조건:**

- 누락 항목의 이름과 필요한 파일 형식을 표시한다.
- 필수 항목이 누락되거나 충돌한 동안 생성 버튼을 비활성화한다.
- 스크립트의 필수 구역 `[제목]`, `[말씀]`, `[기도]` 누락 여부를 확인한다.
- 읽을 수 없거나 빈 파일은 유효한 입력으로 취급하지 않는다.
- 제목·말씀 영상, 기도 영상, 기본 배경음악 및 폰트 등 필수 공통 리소스의 설정 상태를 확인한다.

#### FR-8: 입력 파일 교체 및 제거

사용자는 등록된 입력 파일을 생성 전에 제거하거나 다른 파일로 교체할 수 있다. UJ-2를 실현한다.

**검증 조건:**

- 각 입력 슬롯에서 현재 배정된 파일명을 확인할 수 있다.
- 파일을 제거하면 준비 상태가 즉시 다시 계산된다.
- 새 파일을 등록하면 기존 충돌 또는 누락 안내가 갱신된다.
- 파일을 교체하거나 제거하면 날짜별 작업 폴더의 보관 파일도 일관되게 갱신된다.

#### FR-9: 공통 리소스 설정

사용자는 설정 화면에서 반복 사용하는 제목·말씀 영상, 기도 영상, 기본 배경음악 및 폰트를 확인하고 교체할 수 있다.

**검증 조건:**

- 각 공통 리소스의 현재 파일명과 유효 상태를 표시한다.
- 사용자는 각 공통 리소스를 개별적으로 교체할 수 있다.
- 교체된 공통 리소스는 이후 생성하는 작업부터 적용된다.
- 필수 공통 리소스가 없거나 읽을 수 없으면 메인 화면에서 생성 불가 사유와 설정 화면 이동 동작을 제공한다.
- 기존 완료 영상에는 공통 리소스 변경이 소급 적용되지 않는다.

#### FR-10: 기본 배경음악과 작업별 대체

시스템은 설정된 기본 배경음악을 자동 사용하고, 사용자가 작업별 배경음악을 등록한 경우 해당 작업에만 대체 적용할 수 있다.

**검증 조건:**

- 작업별 배경음악이 없으면 기본 배경음악을 사용한다.
- 작업별 배경음악이 등록되면 기본 배경음악을 변경하지 않고 해당 작업에서만 사용한다.
- 현재 작업에 적용될 배경음악이 기본값인지 작업별 파일인지 메인 화면에서 확인할 수 있다.

### 4.2 쇼츠 자동 생성

**설명:** 준비 상태에서 사용자가 생성 버튼을 누르면 시스템은 `docs/idea.md`에 정의된 제작 규칙을 적용해 영상과 중간 산출물을 생성한다. 사용자가 타임라인을 직접 편집하는 기능은 제공하지 않는다.

#### FR-11: 스크립트 해석

시스템은 스크립트에서 제목, 말씀, 기도 구역과 사용자가 지정한 줄바꿈·자막 블록을 해석할 수 있다.

**검증 조건:**

- `[제목]`, `[말씀]`, `[기도]` 내용을 구분해 구조화한다.
- 기도 구역의 빈 줄과 줄바꿈을 자막 블록 생성 입력으로 보존한다.
- 파싱 결과 또는 파싱 오류를 로그에 기록한다.

#### FR-12: 음성 기반 타이밍 생성

시스템은 음성 녹음을 분석하여 제목·말씀·기도 구간과 자막 타이밍 초안을 생성할 수 있다.

**검증 조건:**

- 녹음 앞부분의 무음을 정리하고 영상 시작 후 약 2초에 음성이 시작되도록 처리한다.
- 분석 결과로 `timing.json`을 생성한다.
- 스크립트의 기도 첫 구절을 음성 분석 결과에서 찾지 못하거나 후보가 여러 개여서 기도 시작점을 하나로 결정할 수 없으면 자동 렌더링을 중단하고 확인 필요 사유를 표시한다.

#### FR-13: 스타일 자막 생성

시스템은 제목, 말씀, 기도 구역별 스타일과 타이밍을 포함한 ASS 자막을 생성할 수 있다.

**검증 조건:**

- `subtitles.ass`를 결과 폴더에 저장한다.
- 제목과 말씀은 상단, 기도는 중앙에 배치한다.
- 폰트, 크기, 색상, 테두리, 그림자 및 필요한 페이드 효과를 적용한다.
- 마지막 “아멘” 자막은 약 2초간 유지하고 약 0.5초 페이드아웃을 적용한다.
- 한글이 깨지지 않고 최종 영상에 표시된다.

#### FR-14: 배경 영상 구성

시스템은 제목·말씀 구간과 기도 구간에 공통 리소스를 적용하여 전체 길이에 맞는 배경 영상을 구성할 수 있다.

**검증 조건:**

- 제목·말씀 영상은 해당 구간 길이에 맞춰 속도를 조정한다.
- 기도 영상은 필요한 길이만큼 반복한다.
- 기도 영상 반복 경계에 약 0.5초 크로스페이드를 적용한다.
- 음성 종료 후 약 3초의 종료 여백을 유지한다.

#### FR-15: 오디오 및 썸네일 합성

시스템은 음성, 배경음악 및 썸네일을 최종 영상 규칙에 맞게 합성할 수 있다.

**검증 조건:**

- 배경음악은 영상 시작부터 재생되고 시작 2초 페이드인과 마지막 2초 페이드아웃이 적용된다.
- 썸네일 이미지는 영상 첫 프레임에 삽입된다.
- 음성과 배경음악이 함께 포함된 최종 오디오가 생성된다.

#### FR-16: 최종 영상 및 중간 산출물 저장

시스템은 생성 결과를 로컬 결과 폴더에 저장할 수 있다.

**검증 조건:**

- 최종 영상은 `1080x1920`, 9:16, MP4 형식으로 생성된다.
- 최종 영상은 30fps를 기본 출력 프레임률로 사용한다.
- `final.mp4`, `subtitles.ass`, `timing.json`, `render_log.txt`를 저장한다.
- 입력 및 결과 파일을 외부 서버로 전송하지 않는다.
- 등록된 입력 파일은 게시 날짜별 작업 폴더에 영구 보관한다.
- 처리에 추가 임시 파일이 필요한 경우 작업 완료·실패·취소 후 삭제한다.
- 드래그앤드롭한 출처 파일은 변경하거나 삭제하지 않는다.

### 4.3 작업 진행 상태

**설명:** 생성 실행 후 메인 화면은 작업의 현재 상태를 음악 재생 화면과 유사한 중심 영역에 표시한다. 진행률은 각 생성 단계의 완료 비중을 기반으로 한 추정치이며 렌더링 완료 시간을 정확히 예측하지 않는다.

#### FR-17: 생성 상태 및 진행률 표시

사용자는 작업이 진행 중인지, 어느 생성 단계인지, 진행률이 어느 정도인지 확인할 수 있다. UJ-1을 실현한다.

**검증 조건:**

- 생성 시작 즉시 제작 중 상태를 표시한다.
- 현재 생성 단계의 이름을 표시한다.
- 0~100% 범위의 진행률을 표시한다.
- 작업이 끝나기 전에 100%로 표시하지 않는다.

#### FR-18: 중복 실행 방지

시스템은 실행 중인 작업이 있을 때 동일 화면에서 다른 작업을 동시에 시작하지 않도록 한다.

**검증 조건:**

- 작업 실행 중 생성 버튼을 비활성화한다.
- 새 파일 등록이 현재 작업 데이터를 손상시키지 않는다.

#### FR-19: 실행 중 작업 취소

사용자는 생성 중인 작업을 취소할 수 있다.

**검증 조건:**

- 생성 중에만 취소 동작을 제공한다.
- 취소 전 확인 절차를 거쳐 실수로 작업을 중단하지 않도록 한다.
- 취소 요청 후 실행 중인 음성 분석 또는 렌더링 프로세스를 안전하게 종료한다.
- 취소된 작업은 완료 항목으로 기록하지 않는다.
- 취소 과정에서 이미 생성된 임시 파일은 정리하고 취소 로그는 보존한다.

### 4.4 완료 및 오류 안내

#### FR-20: 생성 완료 알림

시스템은 작업 완료 시 성공 상태와 결과 위치를 모달로 알릴 수 있다. UJ-1을 실현한다.

**검증 조건:**

- 완료 모달은 최종 영상 생성 성공 여부를 명시한다.
- 사용자는 모달에서 결과 폴더를 열거나 화면으로 돌아갈 수 있다.
- 완료 상태가 메인 화면과 완료 목록에 반영된다.

#### FR-21: 오류 원인 및 다음 동작 안내

시스템은 생성 실패 시 오류가 발생한 생성 단계, 사용자에게 이해 가능한 원인 및 권장 다음 동작을 모달로 표시할 수 있다. UJ-3을 실현한다.

**검증 조건:**

- 단순한 “실패”만 표시하지 않고 오류 범주와 해결 안내를 제공한다.
- 상세 기술 정보는 `render_log.txt`에 기록한다.
- 입력 수정 후 다시 생성할 수 있다.
- 실패한 작업은 완료 항목으로 표시하지 않는다.
- 자동 재시도는 하지 않으며 사용자가 문제 해결 후 생성 버튼을 다시 누른다.

### 4.5 최근 작업 및 완료 목록

**설명:** 메인 화면은 마지막 작업 상태를 중심에 보여주고 오른쪽 영역에 완료 항목을 표시한다. MVP에서는 애플리케이션 내부 영상 재생과 파일 삭제를 제공하지 않는다.

#### FR-22: 최근 작업 표시

시스템은 애플리케이션 실행 시 마지막 작업의 제목, 게시 날짜, 생성 상태, 실제 생성일, 결과 위치를 표시할 수 있다.

**검증 조건:**

- 성공한 작업이 있으면 가장 최근 완료 항목이 기본 선택된다.
- 이력이 없으면 새 작업 등록을 안내하는 빈 상태를 표시한다.

#### FR-23: 완료 목록 표시

사용자는 완료 항목의 제목, 게시 날짜, 실제 생성일을 확인할 수 있다.

**검증 조건:**

- 목록은 애플리케이션을 다시 실행해도 유지된다.
- 각 완료 항목의 제목은 해당 작업의 스크립트 `[제목]` 내용을 한 줄로 정리해 표시한다.
- 각 완료 항목은 대응하는 결과 폴더와 연결된다.
- 게시 날짜를 기준으로 어디까지 제작했는지 확인할 수 있다.
- 완료 목록의 기본 정렬은 게시 날짜 내림차순이다.
- 존재하지 않는 결과 폴더는 정상 완료 항목처럼 열리지 않으며 상태를 안내한다.

#### FR-24: 결과 폴더 열기

사용자는 최근 작업 또는 완료 목록에서 결과 폴더를 운영체제 파일 탐색기로 열 수 있다.

**검증 조건:**

- 한 번의 명시적 동작으로 해당 결과 폴더를 연다.
- 애플리케이션 내부 영상 재생, 다운로드 복제 또는 삭제 기능은 제공하지 않는다.

## 5. 정보 구조 및 화면 방향

MVP는 PC 전용 단일 메인 화면을 기본으로 한다.

- **왼쪽/중앙 주요 영역:** 게시 날짜 선택, 앨범 아트처럼 큰 파일 등록 영역, 입력 슬롯 상태, 생성 버튼
- **재생 상태 영역:** 현재 생성 단계, 진행률, 완료 또는 실패 상태
- **오른쪽 영역:** 제목, 게시 날짜, 실제 생성일을 포함한 완료 목록
- **모달:** 생성 완료, 생성 실패 및 필요한 다음 동작 안내
- **설정 화면:** 제목·말씀 영상, 기도 영상, 기본 배경음악 및 폰트 관리

구체적인 배치, 컴포넌트 상태와 상호작용은 후속 UX 설계에서 확정한다.

## 6. 비기능 요구사항

### 6.1 성능

- 사용자가 게시 날짜 선택과 파일 등록부터 생성 실행까지 수행하는 조작 시간은 정상 입력 기준 5분 이내여야 한다.
- 진행 상태 UI는 생성 단계가 바뀐 뒤 1초 이내에 현재 단계와 진행률을 갱신해야 한다.
- 실제 렌더링 소요 시간은 PC 성능과 영상 길이에 따라 달라지므로 MVP 성공 기준의 5분은 사용자 조작 시간으로 한정한다.
- 시스템은 작업별 실제 렌더링 시간을 로그에 기록한다.

### 6.2 신뢰성 및 복구 가능성

- 실패 시 원인을 추적할 수 있는 로그를 항상 남겨야 한다.
- 입력 파일이나 공통 리소스가 유효하지 않으면 최종 렌더링 전에 실패해야 한다.
- 실패한 작업이 기존 완료 결과를 덮어쓰거나 손상시키지 않아야 한다.
- 결과 파일은 작업별로 구분된 로컬 폴더에 저장해야 한다.
- 등록된 입력 파일은 게시 날짜별 작업 폴더에 보관해야 한다.
- 드래그앤드롭한 출처 파일은 애플리케이션이 삭제하거나 변경하지 않아야 한다.
- 처리용 임시 파일은 작업 종료 후 정리되어 불필요한 저장 공간을 계속 사용하지 않아야 한다.
- 애플리케이션이나 PC가 비정상 종료된 경우 날짜별 입력 파일과 기존 완료 결과를 유지하고, 다음 실행에서 미완료 상태와 재실행 안내를 표시한다. 중단 지점 자동 복구는 MVP에 포함하지 않는다.

### 6.3 개인정보 및 로컬 처리

- 입력 음성, 스크립트, 이미지, 배경음악 및 생성 영상은 사용자가 명시적으로 승인하지 않는 한 외부 서버로 업로드하지 않아야 한다.
- 외부 네트워크 연결이 필요한 음성 분석 서비스는 MVP 기본 동작에 포함하지 않으며 음성 분석은 로컬에서 실행한다.

### 6.4 사용성

- 사용자는 별도의 영상 편집 지식 없이 한 화면에서 작업을 완료할 수 있어야 한다.
- 오류 메시지는 기술 코드보다 사용자가 취할 수 있는 행동을 우선 표시해야 한다.
- 모바일 반응형 UI는 요구하지 않는다.

### 6.5 확장 가능성

- MVP는 개인용 설치형 로컬 애플리케이션으로 제공해야 한다.
- 핵심 영상 생성 규칙은 화면 표시와 분리된 독립적인 생성 기능으로 정의되어야 한다.
- 입력 검증, 생성 실행, 진행 상태 및 결과 정보는 향후 다른 UI나 원격 작업 처리 방식에서도 사용할 수 있는 명확한 경계를 가져야 한다.
- 로컬 전용 구현을 위해 사용자 계정, 클라우드 저장소, 결제 및 다중 사용자 기능을 미리 구축하지 않는다.
- 확장 가능성은 현재 MVP의 설치·실행·유지보수를 불필요하게 복잡하게 만들지 않는 범위에서 확보한다.

## 7. 명시적 비목표

- CapCut과 같은 범용 타임라인 편집기 제공
- 모바일 또는 태블릿 지원
- 여러 사용자 계정, 로그인, 권한 관리
- 클라우드 업로드, 원격 저장 또는 원격 렌더링
- 애플리케이션 내부 영상 재생 및 파일 삭제
- YouTube 업로드 및 예약 발행
- 영상 주제, 기도문 또는 썸네일 자동 생성
- 완전 자동 자막 문장 분할
- 음성 및 배경음악 자동 음량 보정
- 외부 고객을 위한 결제, 구독 및 상품화 기능
- MVP 단계의 클라우드 호스팅 및 서버 렌더링

## 8. MVP 범위

### 8.1 포함

- PC 전용 로컬 UI
- 한 번에 쇼츠 한 개 생성
- 게시 날짜 선택과 날짜별 작업 관리
- 여러 입력 파일의 한 번 드래그앤드롭 등록
- 입력 파일의 날짜별 작업 폴더 보관
- 입력 슬롯 자동 분류, 누락·중복·유효성 안내
- 완료된 게시 날짜의 중복 생성 방지
- 확인 절차를 거친 완료 작업 다시 생성
- 설정 화면에서 공통 배경 영상, 기본 배경음악 및 폰트 관리
- 기본 배경음악 자동 적용과 작업별 배경음악 대체
- 스크립트 파싱과 음성 기반 타이밍 생성
- ASS 자막, 공통 영상, 음성, 배경음악 및 썸네일 합성
- 최종 MP4와 중간 산출물 및 로그 저장
- 생성 단계와 추정 진행률 표시
- 성공·실패 모달과 후속 행동 안내
- 마지막 작업 및 완료 목록
- 결과 폴더 열기

### 8.2 MVP 제외 및 후속 후보

- **6일치 작업 일괄 등록 및 대기열:** 한 개 작업의 품질과 안정성을 먼저 검증한 후 2차에서 고려한다.
- **동시 렌더링:** 로컬 자원 경합과 복잡도를 피하기 위해 제외한다.
- **자동 재시도:** 오류 패턴이 축적된 후 우선순위를 판단한다.
- **YouTube 연동:** 영상 생성 자동화가 검증된 이후 진행한다.
- **서비스화:** 개인 실사용 결과와 상품화 판단 기준이 확보된 이후 별도 제품으로 검토한다.

## 9. 성공 지표

### 9.1 주요 지표

- **SM-1 사용자 작업 시간:** 정상 입력 준비 후 게시 날짜 선택, 파일 등록 및 생성 실행까지의 사용자 조작 시간이 5분 이내다. FR-1~FR-10, FR-17을 검증한다.
- **SM-2 무편집 사용 가능률:** 최소 10개 작업 중 8개 이상이 다음 조건을 모두 충족해 CapCut 등에서 추가 편집 없이 게시 가능해야 한다. FR-11~FR-16을 검증한다.
  - 영상이 `1080x1920`, 30fps MP4로 정상 재생된다.
  - 제목·말씀·기도 자막이 화면 밖으로 잘리거나 한글이 깨지지 않는다.
  - 각 자막 블록은 대응 음성 시작 시점 기준 ±0.75초 안에 표시되고 다음 블록과 잘못 겹치지 않는다.
  - 배경 영상 반복과 전환에 의도하지 않은 검은 프레임이나 정지 구간이 없다.
  - 음성과 배경음악이 모두 들리며 시작·종료 페이드가 적용된다.

### 9.2 보조 지표

- **SM-3 반복 사용 안정성:** 7일 연속으로 하루 한 개 이상의 영상을 생성하며 치명적 데이터 손상 없이 작업을 완료한다. FR-16, FR-20~FR-24를 검증한다.
- **SM-4 오류 해결 가능성:** 사용자 조치가 가능한 오류에는 원인과 다음 동작이 표시되고 로그가 생성된다. FR-7, FR-21을 검증한다.

### 9.3 균형 지표

- **SM-C1 렌더링 속도만을 위해 품질을 낮추지 않는다:** 빠른 생성보다 기존 영상과 유사하고 추가 편집이 필요 없는 결과를 우선한다.
- **SM-C2 자동 분류율을 높이기 위해 불확실한 파일을 임의 배정하지 않는다:** 잘못된 자동 생성보다 충돌 안내와 사용자 교정을 우선한다.

## 10. 위험 및 대응 방향

- **음성 인식과 스크립트 불일치:** 신뢰도가 낮으면 자동 렌더링을 중단하고 확인 필요 사유를 제공한다.
- **파일 역할 충돌:** 확장자만으로 구분하지 못하는 경우 파일명 규칙과 충돌 상태를 사용한다.
- **PC별 렌더링 성능 차이:** 진행률을 추정치로 표현하고 실제 처리 시간을 측정한다.
- **기존 CapCut 결과와의 시각적 차이:** 샘플 완성본을 기준으로 폰트, 자막 위치, 전환 및 오디오를 반복 보정한다.
- **일반 웹 UI의 로컬 파일 접근 제약:** 후속 아키텍처 단계에서 로컬 백엔드 또는 데스크톱 패키징 방식을 결정한다.

## 11. 후속 결정 항목

다음 항목은 MVP 구현을 막지 않으며 지정된 단계에서 결정한다.

- **결과 폴더가 외부에서 삭제된 경우의 기록 정리:** UX 설계에서 누락 상태 표시와 기록 제거 동작을 확정한다.
- **PC 기준 사양별 렌더링 시간 목표:** 아키텍처 및 성능 측정 단계에서 기준 장비와 목표 시간을 확정한다.
- **상품화 판단 신호:** 최소 10개 작업과 7일 연속 실사용 결과를 확보한 뒤 재사용 의향, 설치 난이도와 타인 배포 수요를 평가한다.
