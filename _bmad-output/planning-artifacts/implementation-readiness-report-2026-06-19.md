## PRD 분석

### 기능 요구사항

**FR-1: 게시 날짜 선택**  
사용자는 입력 파일을 등록하고 영상을 생성할 게시 날짜를 선택할 수 있다. 메인 화면에서 게시 날짜를 선택할 수 있어야 하며, 기본값은 현재 날짜이고 다른 날짜로 변경할 수 있어야 한다. 선택 날짜는 입력 상태와 생성 화면에 계속 표시되어야 하고, 날짜가 확정되지 않으면 생성할 수 없어야 한다.

**FR-2: 날짜별 입력 파일 보관**  
시스템은 등록된 입력 파일을 선택한 게시 날짜의 작업 폴더에 복사하여 보관할 수 있다. 게시 날짜별 작업 폴더를 만들고 입력 영역에 파일을 복사하며, 재실행 후에도 등록 파일을 확인할 수 있어야 한다. 같은 날짜에 다시 등록할 때는 묵시적으로 덮어쓰지 않고 교체 여부를 확인해야 하며 출처 파일은 변경하거나 삭제하지 않아야 한다.

**FR-3: 완료된 게시 날짜 중복 방지**  
시스템은 이미 완료된 게시 날짜에 새 영상을 중복 생성하지 않도록 한다. 완료된 날짜임을 표시하고 새 생성 버튼을 비활성화하며, 기존 완료 항목과 결과 폴더를 열 수 있어야 한다. 일반 생성 동작으로 기존 결과를 묵시적으로 덮어쓰지 않아야 한다.

**FR-4: 완료 작업 다시 생성**  
사용자는 이미 완료된 게시 날짜의 입력 파일을 확인·교체한 뒤 명시적으로 영상을 다시 생성할 수 있다. `다시 생성` 동작과 기존 결과 교체 확인 모달을 제공하고, 보관 입력을 그대로 쓰거나 일부 교체할 수 있어야 한다. 새 생성 성공 전까지 기존 영상은 유지되어야 하며, 성공 시 결과를 교체하고 실제 생성일을 갱신한다. 실패하거나 취소되면 기존 영상과 완료 기록을 유지한다.

**FR-5: 다중 파일 드래그앤드롭 등록**  
사용자는 한 작업에 필요한 여러 입력 파일을 한 번의 드래그앤드롭으로 등록할 수 있다. 각 파일의 등록 성공 또는 거부 상태를 표시하고, 슬롯별 개별 클릭 없이 등록할 수 있어야 한다. 지원하지 않는 파일은 다른 파일 등록을 방해하지 않고 별도로 안내해야 한다.

**FR-6: 입력 슬롯 자동 분류**  
시스템은 등록된 파일을 썸네일, 음성, 작업별 배경음악, 스크립트 슬롯으로 자동 분류할 수 있다. 이미지와 텍스트 파일을 각각 후보로 분류하고, 확장자가 겹치는 음성·BGM은 `voice.*`, `bgm.*` 파일명 규칙을 우선 적용한다. 확정 불가능한 파일은 미분류로 표시하고 해결 방법을 안내하며, 같은 슬롯의 복수 후보는 충돌 상태로 표시한다. 작업별 BGM이 없으면 기본 BGM을 사용한다.

**FR-7: 입력 준비 상태 검증**  
시스템은 생성 전에 모든 필수 입력 파일과 공통 리소스의 존재 및 기본 유효성을 검증할 수 있다. 누락 항목과 형식을 표시하고, 누락 또는 충돌 중에는 생성 버튼을 비활성화한다. 스크립트의 `[제목]`, `[말씀]`, `[기도]` 구역을 검증하고 읽을 수 없거나 빈 파일은 거부한다. 제목·말씀 영상, 기도 영상, 기본 BGM, 폰트 등 필수 공통 리소스 상태도 확인한다.

**FR-8: 입력 파일 교체 및 제거**  
사용자는 등록된 입력 파일을 생성 전에 제거하거나 다른 파일로 교체할 수 있다. 각 슬롯에서 파일명을 확인할 수 있고, 제거 즉시 준비 상태가 다시 계산되어야 한다. 새 파일 등록 시 충돌·누락 안내가 갱신되어야 하며 날짜별 작업 폴더의 보관 파일도 일관되게 갱신되어야 한다.

**FR-9: 공통 리소스 설정**  
사용자는 설정 화면에서 반복 사용하는 제목·말씀 영상, 기도 영상, 기본 BGM 및 폰트를 확인하고 교체할 수 있다. 각 리소스의 파일명과 유효 상태를 표시하고 개별 교체를 지원한다. 교체 리소스는 이후 작업부터 적용하며, 필수 리소스가 없거나 읽을 수 없으면 메인 화면에 생성 불가 사유와 설정 이동 동작을 제공한다. 기존 완료 영상에는 소급 적용하지 않는다.

**FR-10: 기본 배경음악과 작업별 대체**  
시스템은 설정된 기본 BGM을 자동 사용하고 작업별 BGM이 등록되면 해당 작업에만 대체 적용할 수 있다. 작업별 BGM이 없으면 기본값을 사용하고, 등록된 경우 기본 설정을 변경하지 않는다. 현재 적용될 BGM이 기본값인지 작업별 파일인지 메인 화면에서 확인할 수 있어야 한다.

**FR-11: 스크립트 해석**  
시스템은 스크립트에서 제목, 말씀, 기도 구역과 사용자가 지정한 줄바꿈·자막 블록을 해석할 수 있다. `[제목]`, `[말씀]`, `[기도]`를 구조화하고, 기도 구역의 빈 줄과 줄바꿈을 자막 블록 입력으로 보존하며 파싱 결과 또는 오류를 로그에 기록한다.

**FR-12: 음성 기반 타이밍 생성**  
시스템은 음성 녹음을 분석하여 제목·말씀·기도 구간과 자막 타이밍 초안을 생성할 수 있다. 앞부분 무음을 정리하고 영상 시작 약 2초 후 음성이 시작되도록 처리하며 `timing.json`을 생성한다. 기도 첫 구절을 찾지 못하거나 복수 후보 때문에 시작점을 하나로 결정할 수 없으면 자동 렌더링을 중단하고 확인 필요 사유를 표시한다.

**FR-13: 스타일 자막 생성**  
시스템은 제목, 말씀, 기도 구역별 스타일과 타이밍을 포함한 ASS 자막을 생성할 수 있다. `subtitles.ass`를 저장하고 제목·말씀은 상단, 기도는 중앙에 배치한다. 폰트, 크기, 색상, 테두리, 그림자와 필요한 페이드를 적용하며 마지막 “아멘”은 약 2초 유지 후 약 0.5초 페이드아웃한다. 한글이 깨지지 않아야 한다.

**FR-14: 배경 영상 구성**  
시스템은 제목·말씀 구간과 기도 구간에 공통 리소스를 적용해 전체 길이에 맞는 배경 영상을 구성할 수 있다. 제목·말씀 영상은 구간 길이에 맞춰 속도를 조정하고, 기도 영상은 필요한 길이만큼 반복하며 경계에 약 0.5초 크로스페이드를 적용한다. 음성 종료 후 약 3초의 종료 여백을 유지한다.

**FR-15: 오디오 및 썸네일 합성**  
시스템은 음성, BGM 및 썸네일을 최종 영상 규칙에 맞게 합성할 수 있다. BGM은 영상 시작부터 재생되어 시작 2초 페이드인과 마지막 2초 페이드아웃이 적용되어야 한다. 썸네일은 첫 프레임에 삽입하고 음성과 BGM이 모두 포함된 최종 오디오를 생성한다.

**FR-16: 최종 영상 및 중간 산출물 저장**  
시스템은 생성 결과를 로컬 결과 폴더에 저장할 수 있다. 최종 영상은 `1080x1920`, 9:16, MP4, 기본 30fps로 생성하고 `final.mp4`, `subtitles.ass`, `timing.json`, `render_log.txt`를 저장한다. 입력과 결과를 외부 서버로 보내지 않고 등록 입력은 날짜별 폴더에 영구 보관한다. 추가 임시 파일은 완료·실패·취소 후 삭제하며 출처 파일은 변경하거나 삭제하지 않는다.

**FR-17: 생성 상태 및 진행률 표시**  
사용자는 작업 진행 여부, 현재 생성 단계와 진행률을 확인할 수 있다. 생성 즉시 제작 중 상태와 단계 이름을 표시하고 0~100% 진행률을 제공하되 작업이 끝나기 전에 100%로 표시하지 않는다.

**FR-18: 중복 실행 방지**  
시스템은 실행 중인 작업이 있을 때 동일 화면에서 다른 작업을 동시에 시작하지 않도록 한다. 실행 중 생성 버튼을 비활성화하고 새 파일 등록이 현재 작업 데이터를 손상시키지 않아야 한다.

**FR-19: 실행 중 작업 취소**  
사용자는 생성 중인 작업을 취소할 수 있다. 생성 중에만 취소를 제공하고 확인 절차를 거쳐야 한다. 음성 분석 또는 렌더링 프로세스를 안전하게 종료하고 취소 작업을 완료 항목에 기록하지 않는다. 임시 파일은 정리하되 취소 로그는 보존한다.

**FR-20: 생성 완료 알림**  
시스템은 작업 완료 시 성공 상태와 결과 위치를 모달로 알릴 수 있다. 최종 영상 성공 여부를 명시하고 결과 폴더 열기 또는 화면 복귀를 제공하며 완료 상태를 메인 화면과 완료 목록에 반영한다.

**FR-21: 오류 원인 및 다음 동작 안내**  
시스템은 실패 단계, 사용자가 이해할 수 있는 원인과 권장 다음 동작을 모달로 표시할 수 있다. 단순 실패만 표시하지 않고 오류 범주와 해결 안내를 제공하며 상세 기술 정보는 `render_log.txt`에 기록한다. 입력 수정 후 재생성할 수 있어야 하며 실패 작업은 완료로 표시하지 않는다. 자동 재시도 없이 사용자가 다시 실행한다.

**FR-22: 최근 작업 표시**  
시스템은 실행 시 마지막 작업의 제목, 게시 날짜, 생성 상태, 실제 생성일, 결과 위치를 표시할 수 있다. 성공 이력이 있으면 가장 최근 완료 항목을 기본 선택하고 이력이 없으면 새 작업 등록 빈 상태를 표시한다.

**FR-23: 완료 목록 표시**  
사용자는 완료 항목의 제목, 게시 날짜, 실제 생성일을 확인할 수 있다. 목록은 재실행 후에도 유지되고 제목은 스크립트 `[제목]` 내용을 한 줄로 표시한다. 각 항목은 결과 폴더와 연결되고 게시 날짜 기준 제작 진척을 확인할 수 있어야 하며 기본 정렬은 게시 날짜 내림차순이다. 결과 폴더가 없으면 정상 완료처럼 열지 않고 상태를 안내한다.

**FR-24: 결과 폴더 열기**  
사용자는 최근 작업 또는 완료 목록에서 한 번의 명시적 동작으로 결과 폴더를 운영체제 파일 탐색기로 열 수 있다. 애플리케이션 내부 영상 재생, 다운로드 복제 또는 삭제 기능은 제공하지 않는다.

**기능 요구사항 합계: 24개**

### 비기능 요구사항

**NFR-1:** 정상 입력에서 게시 날짜 선택과 파일 등록부터 생성 실행까지 사용자의 조작 시간은 5분 이내여야 한다.  
**NFR-2:** 진행 상태 UI는 생성 단계 변경 후 1초 이내에 현재 단계와 진행률을 갱신해야 한다.  
**NFR-3:** 5분 성공 기준은 PC 성능에 좌우되는 렌더링 시간이 아닌 사용자 조작 시간으로 한정한다.  
**NFR-4:** 작업별 실제 렌더링 시간을 로그에 기록해야 한다.  
**NFR-5:** 실패 원인을 추적할 수 있는 로그를 항상 남겨야 한다.  
**NFR-6:** 입력 파일이나 공통 리소스가 유효하지 않으면 최종 렌더링 전에 실패해야 한다.  
**NFR-7:** 실패한 작업이 기존 완료 결과를 덮어쓰거나 손상시키지 않아야 한다.  
**NFR-8:** 결과 파일은 작업별로 구분된 로컬 폴더에 저장해야 한다.  
**NFR-9:** 등록된 입력 파일은 게시 날짜별 작업 폴더에 보관해야 한다.  
**NFR-10:** 드래그앤드롭한 출처 파일을 삭제하거나 변경하지 않아야 한다.  
**NFR-11:** 처리용 임시 파일은 작업 종료 후 정리되어야 한다.  
**NFR-12:** 앱이나 PC가 비정상 종료되어도 날짜별 입력과 기존 완료 결과를 유지하고 다음 실행에서 미완료 상태 및 재실행 안내를 표시해야 한다. 중단 지점 자동 복구는 요구하지 않는다.

### 추가 요구사항 및 제약

- MVP는 PC 전용 단일 사용자 설치형 로컬 애플리케이션이며 한 번에 한 작업만 생성한다.

- PRD의 한 화면 중심 PC 경험은 사이드바, 좌측 작업 영역, 우측 완료 목록, 하단 상태 바로 구체화되며 모바일 비대상 범위와 일치한다.
- 원본 불변, 날짜별 사본, 임시 파일 정리, 기존 결과 보호와 외부 네트워크 불필요 원칙이 UX의 로컬 파일 안전 계약에 반영되어 있다.
- 결과 폴더가 외부에서 삭제된 경우 경고 상태는 UX에 정의되어 있으나 기록 제거 정책은 PRD와 UX 모두 후속 결정으로 남아 있다.

### UX ↔ 아키텍처 정렬

- Electron main/preload/renderer 분리는 시스템 파일 선택기, Explorer/Finder 연동과 최소 권한 로컬 파일 접근을 지원한다.
- `job-editor`, `job-progress`, `job-history`, `resources`, `guide` 기능 경계가 UX의 홈, 진행 상태, 완료 목록, 설정 모달, 가이드 화면과 직접 대응한다.
- Zustand 상태 머신은 준비, 실행, 취소 중, 성공, 실패, 중단과 재생성 상태를 구분하고 실행 스냅샷을 보호한다.

- 전역 CSS custom properties, CSS Modules, 접근성 공유 영역, 1180×720 최소 창, 키보드·포커스·200% 확대 요구가 아키텍처와 에픽에 반영되어 있다.
- SQLite와 실제 결과 경로 대조는 완료 목록과 결과 폴더 누락 상태를 지원한다.

#### 1. Story 2.1의 최소 수직 슬라이스 산출물이 명확하지 않음

Story 2.1은 `start → stage_started → progress → completed` 실제 엔진 흐름을 요구하지만 완료 시 생성되는 최소 사용자 산출물 또는 검증 가능한 파일을 명시하지 않는다. Stories 2.2~2.9가 파싱, 음성 분석, 자막, 합성과 결과 커밋을 후속 구현하므로 현재 조건만으로는 실제 수직 슬라이스인지 프로토콜 스텁인지 완료 판정이 모호하다.

**권고:** Story 2.1에 라이선스 안전 샘플로 생성하는 최소 MP4 또는 명시적인 진단 산출물과 완료 검증을 추가한다. 그렇지 않으면 제목을 “엔진 실행 계약과 진행 상태 기반”으로 바꾸고 사용자 가치가 아닌 기술 기반 스토리임을 명확히 관리한다.

#### 2. 일부 SQLite 스키마 변경의 최초 소유 스토리가 여전히 불명확함

Story 1.3은 `jobs` 기반 마이그레이션을, Story 2.1은 `job_attempts` 마이그레이션을 명시한다. 그러나 Story 1.4의 입력 메타데이터와 Story 1.7의 리소스 메타데이터에 필요한 `job_inputs`, `resources` 테이블 또는 열·인덱스의 생성 시점이 인수 조건에 명시되지 않았다.

**권고:** Story 1.4와 Story 1.7에 각 기능이 최초 사용하는 테이블·열·제약·인덱스를 순차 마이그레이션으로 추가한다는 조건을 명시한다.
