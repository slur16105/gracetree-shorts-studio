# Sprint Change Proposal

## 1. 이슈 요약

완성 영상을 사용자 다운로드 폴더로 자동 복사하는 기능(`GTS_YYMMDDHHMM.mp4`)이 구현되면서, 앱의 데이터 보관 모델이 더 이상 기존 PRD·아키텍처와 일치하지 않는다. 사용자는 다음 세 가지 변경을 요청했다.

1. **완료 목록 비영구화** — 완료 작업 목록을 SQLite에 영구 저장하지 않고, 앱 실행~종료(세션) 동안만 표시한다. 다음 실행 시 완료 목록은 비어 있다.
2. **"폴더 열기" 재지정(다운로드 폴더)** — 완성 영상이 다운로드 폴더로 나가므로, 각 결과 폴더를 여는 대신 **모든 "열기" 동작이 다운로드 폴더를 열도록 재지정**한다. (결과 폴더는 세션 종료 시 삭제되어 열 의미가 없다.) 버튼은 유지하되 대상만 바꾼다.
3. **제작 소스 비보관** — 제작에 쓴 입력/중간 산출물(`input/`, `temp/`, `output/`)을 영구 보관하지 않는다. 현재는 영구 보관 중이다.
4. **완료 목록 표시 보정** — 완료 목록 각 행에서 **생성일(completedAt)을 제거하고 그 자리에 제목을 표시**한다. 제목은 스크립트 한 줄 요약에서 추출되지만 현재 DB에 저장되지 않아 비어 있으므로, **생성 시작 시 제목을 job에 저장**한다.

세 변경의 공통 의도는 앱을 **"보관·이력 모델"에서 "내보내고 잊는(export-and-forget) 모델"로 전환**하는 것이다. 완성 영상은 다운로드 폴더에 남고, 작업용 데이터는 세션이 끝나면 정리된다.

### 사용자와 확정한 핵심 설계 결정

- **작업 데이터 수명 = 앱 세션 수명.** `input/temp/output`은 실행 중 유지되고, 앱 종료 시 삭제하며, 다음 시작 시 이전 세션 찌꺼기를 일괄 청소(sweep)한다. (종료-시-삭제만으로는 강제 종료·크래시를 못 막으므로 시작-시-청소를 병행한다.)
- **교차 세션 재생성은 미지원.** 사용자가 원본 소스를 직접 보유하므로, 다시 만들려면 원본을 재등록한다. 세션 내 재생성(Story 3.3)은 데이터가 살아있는 동안 그대로 동작한다.
- **삭제 안전선 = 복사 시점 검증.** 다운로드 복사 성공 여부는 **복사하는 순간**에 판정해 작업에 "내보내기 완료" 플래그로 기록한다. 삭제는 이 플래그만 보고 결정하며, **다운로드 폴더를 재확인하지 않는다.** (사용자가 세션 중 다운로드 파일을 옮기거나 지워도 오판하지 않기 위함. 복사가 애초에 실패한 작업은 플래그가 없어 삭제되지 않고 `output/`이 보존된다.)

## 2. 영향 분석

### Epic 영향

- **Epic 4 (완료 이력 관리)** — 가장 크게 영향. "이력"의 의미가 영구 보관에서 세션 한정으로 축소된다. Epic 목표 문장 재작성 필요.
- **Epic 1 (생성 가능한 작업 준비)** — 입력 파일 "영구 보관" 약속이 "세션 보관 후 정리"로 바뀐다.
- **Epic 2 (생성과 완료)** — 완료 UI에서 폴더 열기 제거.
- **Epic 3 (안정성·복구)** — 재생성(Story 3.3)이 세션 한정으로 의미 축소.

### 산출물 충돌

- **PRD** — FR-2(입력 보관), FR-4(완료 작업 다시 생성), FR-16(최종 산출물 영구 보관), FR-22(최근 작업), FR-23(완료 목록 영속성), FR-24(결과 폴더 열기), FR-20(완료 알림 모달의 폴더 열기), NFR-7(로컬 저장소 구분)과 충돌. NFR-8(원본 보존)·NFR-9(임시 파일 정리)는 유지(오히려 강화).
- **Architecture** — "SQLite는 완료 이력의 유일한 기준", 사용자 데이터 디렉터리 생명주기(`input`/`temp`/`output`), preload API 목록의 "결과 폴더 열기", `files/open-folder.ts` 결정과 충돌.
- **UX** — UX-DR12(폴더 열기 동작), UX-DR20(결과 폴더 없을 때 처리)과 충돌.

### 기술 영향 (코드·DB)

- **DB**: jobs 테이블에 "내보내기 완료(exported)" 상태/플래그 컬럼 추가 마이그레이션 필요. 시작-시-청소가 이전 세션 jobs/job_inputs 행을 정리.
- **Electron main**: 앱 종료 훅(`before-quit`/`will-quit`)과 시작 시 sweep 추가. `openResultFolder` IPC·`open-folder.ts` 제거. 다운로드 복사 성공 시 exported 플래그 기록(현재 best-effort 복사에 검증·기록 추가).
- **Renderer**: ResultDialog·CompletionList에서 폴더 열기 버튼/핸들러 제거. 완료 목록을 세션 메모리 기준으로 전환.
- **contracts**: `openResultFolder` 채널·DTO 제거, completed-jobs 조회 계약 정리.
- **이미 구현한 다운로드 복사 모듈**(`download-export.ts`): 복사 후 성공 검증(크기 대조) 추가, 결과를 호출부에 반환해 플래그 기록에 사용.

## 3. 권고 접근

**MVP 범위 재정의 (Direct Adjustment + 부분 Rollback).** 전체 재기획은 불필요하다. PRD/아키텍처의 데이터 보관 관련 조항만 "세션 수명 + 내보내고 잊기" 모델로 교체하고, 완료된 5개 스토리(4.1, 2.10, 1.4, 2.9, 3.3)에 보정 변경을 적용한다. 폴더 열기 관련 코드는 제거(rollback)한다.

- **근거**: 세 변경은 서로 얽혀 하나의 일관된 모델로 수렴한다. 개별 패치보다 단일 정책 전환이 충돌을 줄인다.
- **위험**: 이 변경은 **파괴적(데이터 삭제 도입)**이다. 삭제 안전선(복사 시점 검증→플래그→플래그 기준 삭제)을 회귀 테스트로 못 박지 않으면 영상 유실 위험이 있다. 이 안전선이 가장 중요한 인수 조건이다.
- **효과**: 디스크 사용량 감소, "파일을 못 찾는" 불편 해소, 모델 단순화. FR-4(교차 세션 재생성) 상실은 사용자가 원본 보유로 수용함.

```
- 앱 종료 훅: exported된 작업 디렉터리 삭제 (output은 exported 가드)
- 앱 시작 sweep: 이전 세션 jobs/job_inputs + 작업 디렉터리 정리
- 회귀 테스트: 복사 실패 시 보존, 강제 종료 후 시작 sweep, exported 가드, 다운로드 파일 이동 후에도 오판 없음

신규 Story (Epic 2 또는 4): "완료 목록 제목 표시와 다운로드 폴더 열기"
- start_job 명령에 title을 실어 엔진이 jobs.title을 저장 (계약·스키마 갱신)
- 완료 목록 행: 게시 날짜 + 제목 표시, 생성일 제거
- openResultFolder(jobId) → openDownloadsFolder() 전환, 3곳의 열기 버튼(JobEditor·CompletionList·ResultDialog) 재지정
- 테스트: 제목 저장·표시, 빈 제목 폴백, 다운로드 폴더 열기
```

## 5. 구현 핸드오프

**변경 범위 분류: Major.** PRD FR 다수 변경, 아키텍처 데이터 보관 불변조건 교체, DB 마이그레이션 신규, 파괴적 삭제 도입.

권장 순서:
1. **PM/Architect**: PRD(§4.1)·Architecture(§4.2) 갱신 — 데이터 보관 모델을 "세션 수명 + 내보내고 잊기"로 확정.
2. **PO/SM**: epics.md(Epic 1·2·4)와 영향 스토리(§4.4) 보정, 신규 스토리(§4.5) 추가, 스프린트 재계획.
3. **Dev**: 신규 스토리 TDD 구현 — 삭제 안전선(복사 시점 검증→exported 플래그→플래그 기준 삭제) 회귀 테스트를 최우선 인수 조건으로.

**성공 기준**
- 앱 종료/재시작 후 완료 목록이 비고, 이전 세션 작업 디렉터리가 정리됨.
- 결과 화면·완료 목록에 폴더 열기 버튼/IPC가 없음.
- 완성 영상은 항상 다운로드 폴더에 `GTS_YYMMDDHHMM.mp4`로 존재.
- **복사 실패 작업의 output은 절대 삭제되지 않음** (영상 유실 0건).
- 다운로드 파일을 세션 중 이동/삭제해도 정리 로직이 오판하지 않음.
