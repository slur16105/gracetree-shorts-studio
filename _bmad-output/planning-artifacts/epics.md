# gracetree-shorts-studio - Epic Breakdown

## Overview

이 문서는 PRD, UX Design 및 Architecture 요구사항을 구현 가능한 에픽과 스토리로 분해하기 위한 전체 요구사항 인벤토리를 제공한다.

## Requirements Inventory

### Functional Requirements

FR1: 사용자는 입력 파일을 등록하고 영상을 생성할 게시 날짜를 선택할 수 있다.

FR2: 시스템은 등록된 입력 파일을 선택한 게시 날짜의 앱 관리 작업 폴더에 복사하고 재실행 후에도 유지할 수 있다.

FR3: 시스템은 이미 완료된 게시 날짜의 일반 중복 생성을 차단하고 기존 완료 항목과 결과 폴더 접근을 제공할 수 있다.

FR4: 사용자는 완료된 게시 날짜의 입력을 확인·교체한 뒤 명시적 확인을 거쳐 다시 생성할 수 있으며, 새 생성 성공 전까지 기존 결과를 보존할 수 있다.

FR5: 사용자는 한 작업에 필요한 여러 입력 파일을 한 번의 드래그앤드롭 또는 동등한 파일 선택 동작으로 등록할 수 있다.

FR6: 시스템은 등록 파일을 썸네일, 음성, 작업별 배경음악, 스크립트 슬롯으로 자동 분류하고 미분류·충돌 상태를 안내할 수 있다.

FR7: 시스템은 생성 전에 필수 입력 파일, 스크립트 구역 및 공통 리소스의 존재와 기본 유효성을 검증하고 준비되지 않은 생성을 차단할 수 있다.

FR8: 사용자는 생성 전에 입력 슬롯의 파일을 제거하거나 교체할 수 있으며 시스템은 보관 파일과 준비 상태를 즉시 일관되게 갱신할 수 있다.

FR9: 사용자는 설정 화면에서 제목·말씀 영상, 기도 영상, 기본 배경음악 및 자막 폰트의 현재 상태를 확인하고 교체할 수 있다.

FR10: 시스템은 작업별 배경음악이 없으면 기본 배경음악을 사용하고, 작업별 배경음악이 있으면 해당 작업에만 대체 적용할 수 있다.

FR11: 시스템은 스크립트의 `[제목]`, `[말씀]`, `[기도]` 구역과 사용자가 지정한 줄바꿈·자막 블록을 구조화해 해석할 수 있다.

FR12: 시스템은 로컬에서 음성을 분석해 제목·말씀·기도 구간과 자막 타이밍을 생성하고, 기도 시작점을 신뢰성 있게 결정할 수 없으면 렌더링을 중단할 수 있다.

FR13: 시스템은 구역별 위치·스타일·타이밍과 마지막 “아멘” 유지·페이드를 포함한 한글 ASS 자막을 생성할 수 있다.

FR14: 시스템은 제목·말씀 영상의 속도를 구간 길이에 맞추고 기도 영상을 크로스페이드와 함께 반복해 전체 배경 영상을 구성할 수 있다.

FR15: 시스템은 음성, 시작·종료 페이드가 적용된 배경음악 및 첫 프레임 썸네일을 최종 영상에 합성할 수 있다.

FR16: 시스템은 `1080x1920`, 9:16, 30fps MP4와 `subtitles.ass`, `timing.json`, `render_log.txt`를 로컬 결과 폴더에 저장하고 임시 파일을 정리할 수 있다.

FR17: 사용자는 생성 중 현재 상태, 생성 단계 및 0~100% 진행률을 확인할 수 있다.

FR18: 시스템은 실행 중인 작업이 있을 때 다른 작업의 동시 실행을 차단하고 현재 작업 데이터를 보호할 수 있다.

FR19: 사용자는 확인 절차를 거쳐 실행 중 작업을 취소할 수 있으며 시스템은 하위 프로세스와 임시 파일을 정리하고 완료 기록을 만들지 않을 수 있다.

FR20: 시스템은 생성 성공 시 결과 위치를 포함한 완료 모달을 표시하고 메인 화면과 완료 목록을 갱신할 수 있다.

FR21: 시스템은 생성 실패 시 실패 단계, 이해 가능한 원인 및 권장 다음 동작을 표시하고 기술 세부정보를 로그에 남길 수 있다.

FR22: 시스템은 앱 실행 시 마지막 작업의 제목, 게시 날짜, 상태, 실제 생성일 및 결과 위치를 표시할 수 있다.

FR23: 사용자는 재실행 후에도 유지되는 완료 목록에서 제목, 게시 날짜, 실제 생성일 및 결과 폴더 상태를 확인할 수 있다.

FR24: 사용자는 최근 작업 또는 완료 목록에서 결과 폴더를 운영체제 파일 탐색기로 열 수 있다.

### NonFunctional Requirements

NFR1: 정상 입력에서 게시 날짜 선택부터 파일 등록과 생성 실행까지의 사용자 조작 시간은 5분 이내여야 한다.

NFR2: 생성 단계 변경 후 1초 이내에 UI의 현재 단계와 진행률이 갱신되어야 한다.

NFR3: 시스템은 작업별 실제 렌더링 시간을 진단 로그에 기록해야 한다.

NFR4: 모든 실패와 생성 시도는 원인을 추적할 수 있는 로그를 남겨야 한다.

NFR5: 유효하지 않은 입력 또는 공통 리소스는 최종 렌더링 전에 검출되어야 한다.

NFR6: 실패·취소·비정상 종료 및 재생성 실패는 기존 완료 결과와 완료 기록을 손상시키지 않아야 한다.

NFR7: 등록 입력과 결과는 작업별로 구분된 로컬 폴더에 저장되어야 한다.

NFR8: 드래그앤드롭 또는 파일 선택으로 받은 출처 원본은 수정하거나 삭제하지 않아야 한다.

NFR9: 처리용 임시 파일은 성공·실패·취소 후 정리되어야 한다.

NFR10: 비정상 종료 후 날짜별 입력과 기존 완료 결과를 유지하고 다음 실행에서 미완료 상태와 재실행 안내를 제공해야 한다.

NFR11: 입력 음성, 스크립트, 이미지, 배경음악 및 생성 영상은 명시적 승인 없이 외부 서버로 전송되지 않아야 한다.

NFR12: 음성 분석을 포함한 MVP 기본 동작은 외부 네트워크 연결 없이 로컬에서 실행되어야 한다.

NFR13: 사용자는 별도의 영상 편집 지식 없이 한 화면의 핵심 흐름으로 작업을 완료할 수 있어야 한다.

NFR14: 사용자 오류 메시지는 기술 코드보다 원인 범주와 사용자가 취할 행동을 우선해야 한다.

NFR15: 제품은 Windows와 macOS에서 동작하는 개인용 설치형 로컬 애플리케이션이어야 하며 모바일 반응형 UI를 요구하지 않는다.

NFR16: 핵심 영상 생성 규칙은 UI 및 저장 방식과 분리된 독립 경계를 가져야 한다.

NFR17: 입력 검증, 생성 실행, 진행 상태 및 결과 정보는 향후 다른 UI나 원격 처리 방식에서도 사용할 수 있는 명확한 계약을 가져야 한다.

NFR18: MVP 확장성 확보를 이유로 계정, 클라우드 저장소, 결제, 다중 사용자 또는 분산 인프라를 미리 구현하지 않아야 한다.

NFR19: 최소 10개 작업 중 8개 이상은 추가 편집 없이 게시 가능한 영상·자막·전환·오디오 품질 기준을 충족해야 한다.

NFR20: 자막 블록은 대응 음성 시작 시점 기준 ±0.75초 안에 표시되고 다음 블록과 잘못 겹치지 않아야 한다.

NFR21: 시스템은 7일 연속 하루 한 개 이상의 영상을 치명적 데이터 손상 없이 생성할 수 있어야 한다.

NFR22: UI는 WCAG 2.2 AA를 목표로 하며 일반 텍스트 4.5:1, 큰 텍스트와 비텍스트 UI 3:1 이상의 대비를 제공해야 한다.

NFR23: 핵심 흐름은 키보드로 조작 가능해야 하고 200% 디스플레이 확대에서도 주요 작업이 가능해야 한다.

NFR24: 최종 영상 품질은 자동 테스트뿐 아니라 기준 영상과의 수동 시각·청각 비교도 통과해야 한다.

### Additional Requirements

- **첫 구현 스토리 필수:** `npm create @quick-start/electron@latest apps/desktop -- --template react-ts`로 Electron + electron-vite + React/TypeScript 스타터를 초기화하고 pnpm 워크스페이스, 공유 계약 패키지 및 Python 엔진 패키지의 기반 구조를 만든다.
- Node 패키지 관리자는 pnpm 11.8 계열을 사용하고 실제 구현 버전은 lockfile로 고정한다.
- renderer, preload, main은 TypeScript, UI는 React, 데스크톱 런타임은 Electron을 사용한다.
- Python 생성 엔진은 별도 패키지 경계에 두고 renderer가 파일 시스템이나 프로세스 API를 직접 호출하지 않도록 한다.
- Electron renderer는 Node.js 직접 접근을 금지하고 `contextIsolation`, sandbox, 로컬 자원만 허용하는 CSP를 적용한다.
- preload는 파일 선택, 작업 조회·실행·취소, 공통 리소스 관리, 결과 폴더 열기에 필요한 좁고 타입이 지정된 API만 노출한다.
- 파일 경로, 작업 ID 및 명령 인자는 main과 Python 경계에서 재검증하며 하위 프로세스는 셸 문자열이 아닌 실행 파일과 인자 배열로 실행한다.
- SQLite를 작업 상태와 완료 이력의 유일한 메타데이터 저장소로 사용하고 Python 표준 `sqlite3`를 통해 접근한다.
- SQLite에는 내부 작업 ID, 게시 날짜, 제목, 상태, 입력·결과 경로, 생성일, 실패 단계 및 오류 범주를 저장한다.
- 게시 날짜 중복은 DB 제약과 애플리케이션 검증을 함께 사용하고 스키마 변경은 순차 번호의 명시적 마이그레이션으로 관리한다.
- 앱 시작 시 SQLite 기록과 실제 파일을 대조해 결과 폴더 누락과 중단 작업을 식별하고 실행 중 상태는 `interrupted`로 정리한다.
- 재생성은 작업별 임시 위치에 새 결과를 만든 뒤 검증과 DB 트랜잭션이 성공한 경우에만 기존 결과를 교체한다.
- Electron main과 장기 실행 Python 엔진은 stdin/stdout JSON Lines 프로토콜로 통신하고 stderr는 진단 로그로 분리한다.
- 모든 프로토콜 메시지는 `protocolVersion`, `type`, `jobId`, `timestamp`, `payload`를 포함하고 TypeScript와 Python 양쪽에서 동일한 JSON Schema로 검증한다.
- 프로토콜은 `start`, `cancel` 명령과 `accepted`, `stage_started`, `progress`, `artifact_created`, 최종 상태 이벤트를 지원해야 한다.
- 한 작업에서는 `completed`, `failed`, `cancelled` 중 하나의 최종 이벤트만 허용하고 진행률은 단조 증가하며 완료 전에 100%가 될 수 없다.
- 취소 시 구조화된 명령으로 정상 종료를 먼저 시도하고 응답하지 않으면 FFmpeg와 음성 분석 하위 프로세스를 포함한 프로세스 트리를 종료한다.
- React UI 상태는 Zustand 5.0.14 계열과 순수 전이 함수를 사용하는 타입 기반 작업 상태 머신으로 관리한다.
- 작업 상태는 `idle`, `validating`, `analyzing`, `subtitling`, `composing`, `saving`, `completed`, `failed`, `cancelling`, `cancelled`, `interrupted`를 포함하고 허용된 전이만 적용한다.
- Python 이벤트는 이벤트 어댑터를 통해 UI 도메인 이벤트로 변환하며 현재 작업과 다른 `jobId`의 이벤트는 적용하지 않는다.
- 실행 중인 작업 스냅샷과 다음 작업을 위한 입력 편집 상태를 분리한다.
- Python 엔진은 PyInstaller 6.21 계열 onedir 배포물로 생성하며 FFmpeg와 faster-whisper 1.2.1 계열의 로컬 모델을 ASAR 외부 리소스로 포함한다.
- Windows x64와 macOS arm64 배포물은 각 운영체제 빌드 환경에서 electron-builder로 생성하고 포함 바이너리의 버전·체크섬·라이선스를 기록한다.
- 사용자에게 Python, FFmpeg 또는 별도 Python 패키지 설치를 요구하지 않아야 한다.
- 외부 배포 전 코드 서명과 macOS notarization을 릴리스 게이트로 적용하며 자동 업데이트는 MVP에서 제외한다.
- Vitest와 React Testing Library로 TypeScript 단위·컴포넌트 테스트, pytest로 Python 엔진 테스트, 공유 픽스처로 양방향 프로토콜 계약 테스트를 수행한다.
- 작은 라이선스 안전 샘플로 스크립트 파싱부터 FFmpeg 렌더링까지 통합 테스트하고 Playwright로 등록·준비·진행·실패·취소·완료 흐름을 검증한다.
- Windows/macOS 설치본에서 앱 시작, 사용자 데이터 초기화, SQLite 마이그레이션, 포함 바이너리 실행 및 샘플 렌더링 스모크 테스트를 수행한다.
- 작업별 로그에는 `attemptId`, 단계 시작·종료, 처리 시간, 민감 정보를 제거한 실행 정보, 오류 코드 및 진단을 기록한다.
- 안정적인 오류 코드 집합(`INPUT_MISSING`, `INPUT_CONFLICT`, `SCRIPT_INVALID`, `PRAYER_BOUNDARY_AMBIGUOUS`, `RESOURCE_INVALID`, `PROCESS_FAILED`, `OUTPUT_COMMIT_FAILED`, `JOB_INTERRUPTED`)을 공유 계약으로 관리한다.
- 사용자 데이터는 `studio.db`, `resources/`, 게시 날짜별 `jobs/<date>/input|output|temp|logs` 구조를 사용하고 최종 경로에 직접 렌더링하지 않는다.
- 데이터베이스/Python은 `snake_case`, JSON/TypeScript는 `camelCase`, 게시 날짜는 `YYYY-MM-DD`, 시각은 UTC ISO 8601을 사용하며 경계 어댑터에서 명시적으로 변환한다.
- 기준 Windows/macOS 장비의 렌더링 성능 예산과 faster-whisper 모델 크기·양자화·CPU thread 설정은 첫 수직 슬라이스의 실제 샘플 측정으로 확정한다.

### UX Design Requirements

UX-DR11: `Primary Button`은 준비 완료 시에만 활성화하고 버튼 묶음 오른쪽에 배치하며, `Secondary Button`은 주요 동작보다 먼저 배치해야 한다.

UX-DR13: `Bottom Status Bar`는 유휴 시 최근 작업, 실행 중에는 단계와 0~99% 진행률을 표시하고 입력 준비율과 렌더링 진행률을 혼합하지 않아야 한다.

UX-DR15: `Guide Navigation`과 `Guide Card`는 첫 영상 만들기, 파일명 규칙, 스크립트 작성법, 오류 해결, 저장 위치 및 앱 정보를 제공하고 홈 복귀 시 입력 상태를 보존해야 한다.

UX-DR16: `Result Dialog`는 성공·실패·취소·재생성 확인에 맞는 한 문장 요약과 다음 동작을 제공하고 원시 기술 로그를 직접 노출하지 않아야 한다.

UX-DR17: 첫 실행, 파일 등록 중, 입력 누락, 입력 충돌, 제목 누락, 공통 리소스 누락, 준비 완료 상태마다 명확한 빈 상태·안내·활성화 규칙을 구현해야 한다.

UX-DR18: 생성 중에는 실행 스냅샷을 보호하고 취소 요청 중, 성공, 실패, 비정상 종료 후 재실행 상태를 서로 구분해 표시해야 한다.

UX-DR19: 완료 날짜 선택과 다시 생성 흐름은 일반 생성을 차단하고 기존 결과 유지 조건과 명시적 확인을 사용자에게 알려야 한다.

UX-DR20: 결과 폴더가 없을 때 정상 완료처럼 열지 않고 `결과 폴더를 찾을 수 없습니다` 상태를 표시해야 한다.

UX-DR21: 마우스·트랙패드와 키보드를 동등하게 지원하고 `Tab`, `Enter/Space`, `Esc`의 정의된 상호작용을 제공해야 한다.

UX-DR23: 생성 진행률은 단계 변경 또는 10% 단위 통과 시에만 polite live region으로 알리고 연속 갱신을 반복 낭독하지 않아야 한다.

UX-DR24: 파일 배치 등록은 처리 완료 후 성공·거부 개수를 한 번 알리고 개별 오류를 대응 File Slot과 연결해야 한다.

UX-DR26: Windows에서는 시스템 파일 선택기와 Explorer, macOS에서는 시스템 파일 선택기와 Finder를 사용하고 플랫폼 경로 문법을 UI 문구에 하드코딩하지 않아야 한다.

## Epic 1: 생성 가능한 작업 준비

사용자는 게시 날짜와 입력 파일을 안전하게 등록·분류·검증하고 반복 사용하는 공통 리소스를 설정하여 영상 생성 준비를 완료할 수 있다.

### Story 1.1: 프로젝트 스타터와 계약 기반 초기화

As a 개발 담당자,
I want 승인된 스타터와 프로젝트 경계를 초기화하고 싶다,
So that 이후 기능을 일관된 구조와 검증 가능한 계약 위에서 구현할 수 있다.

**Requirements:** Architecture starter requirement

**Acceptance Criteria:**

**Given** 개발 환경에 Node.js와 pnpm이 설치되어 있을 때
**When** `npm create @quick-start/electron@latest apps/desktop -- --template react-ts`와 루트 워크스페이스 초기화를 실행하면
**Then** Electron, electron-vite, React, TypeScript 기반 앱이 실행된다.
**And** `apps/desktop`, `packages/contracts`, `engine` 패키지 구조가 생성된다.

**Given** Python 엔진 개발 진입점을 실행했을 때
**When** 상태 확인용 JSON Lines 메시지를 주고받으면
**Then** 유효한 프로토콜 envelope로 응답한다.
**And** 일반 진단 로그는 stdout 프로토콜과 분리된다.

**Given** 프로젝트 검증 명령을 실행할 때
**When** 타입 검사와 기본 테스트가 수행되면
**Then** TypeScript 워크스페이스와 공유 계약 테스트가 통과한다.
**And** Python 엔진의 기본 프로토콜 테스트가 통과한다.

### Story 1.2: 안전하고 접근 가능한 데스크톱 앱 셸

As a 개인 제작자,
I want Windows와 macOS에서 실행되는 안전한 데스크톱 앱 셸을 사용하고 싶다,
So that 영상 제작 기능을 일관된 로컬 환경에서 사용할 수 있다.

**Requirements:** NFR13, NFR15, NFR18, NFR22, NFR23; UX-DR1-UX-DR5, UX-DR21, UX-DR25, UX-DR27, UX-DR28, UX-DR30

**Acceptance Criteria:**

**Given** 사용자가 앱을 실행했을 때
**When** 메인 창이 표시되면
**Then** 홈, 사용 가이드, 설정 진입점이 있는 기본 앱 셸이 표시된다.
**And** 1180×720의 권장 레이아웃과 다크 디자인 토큰이 적용된다.

**Given** renderer가 운영체제 기능을 사용해야 할 때
**When** 파일이나 프로세스 기능에 접근하려 하면
**Then** renderer는 Node.js API에 직접 접근할 수 없다.
**And** `contextIsolation`, sandbox 및 로컬 리소스 전용 CSP가 활성화된다.

**Given** renderer와 main 프로세스가 통신할 때
**When** preload API를 사용하면
**Then** 타입이 지정된 허용 API만 노출된다.
**And** 임의 파일 시스템 및 프로세스 실행 기능은 노출되지 않는다.

**Given** 사용자가 키보드만 사용할 때
**When** `Tab`, `Enter`, `Space`로 기본 앱 셸을 탐색하면
**Then** 모든 전역 진입점에 접근할 수 있다.
**And** 현재 포커스와 활성 화면이 색상 외의 방식으로도 구분된다.

**Given** 앱 셸의 상태와 선택 지점을 표시할 때
**When** 디자인 토큰을 적용하면
**Then** 브랜드 분홍색은 선택, 녹색은 준비·진행·성공, 경고와 위험 색상은 확인 필요·실패 의미로 사용한다.
**And** 일반 텍스트 4.5:1, 큰 텍스트와 비텍스트 UI 3:1 이상의 목표 대비를 유지한다.

**Given** 운영체제에서 모션 감소가 활성화되어 있을 때
**When** 앱 셸과 상태 피드백을 표시하면
**Then** 반복 펄스와 불필요한 이동 애니메이션을 줄인다.
**And** 상태 의미를 애니메이션에만 의존하지 않는다.

**Given** 창이 1180×720의 권장 크기보다 작아질 때
**When** 핵심 레이아웃을 표시하면
**Then** 모바일 스택으로 재배치하지 않는다.
**And** 주요 버튼과 진행률이 잘리지 않도록 최소 창 크기를 제한하거나 안내한다.

**Given** 네트워크 연결이 없을 때
**When** 앱을 실행하고 기본 화면을 탐색하면
**Then** 앱 셸과 로컬 기능은 정상 동작한다.
**And** 외부 연결을 요구하는 오류나 배너를 표시하지 않는다.

**Given** Windows 또는 macOS의 디스플레이 배율이 200%일 때
**When** 홈, 가이드 및 설정 화면을 탐색하면
**Then** 핵심 정보와 주요 동작을 확대·스크롤을 통해 사용할 수 있다.

### Story 1.3: 게시 날짜별 작업 생성과 복원

As a 개인 제작자,
I want 쇼츠의 게시 날짜를 선택하고 날짜별 작업을 보존하고 싶다,
So that 앱을 다시 실행해도 준비하던 작업을 이어갈 수 있다.

**Requirements:** FR1, FR2; NFR7, NFR10; UX-DR8, UX-DR17, UX-DR18

**Acceptance Criteria:**

**Given** 사용자가 홈 화면을 처음 열었을 때
**When** 게시 날짜 선택기가 표시되면
**Then** 현재 날짜가 기본값으로 선택된다.
**And** 날짜는 `YYYY-MM-DD · Day` 형식으로 표시된다.

**Given** 날짜 선택기에 포커스가 있을 때
**When** 사용자가 키보드로 달력을 조작하면
**Then** 화살표, Home/End, PageUp/PageDown, Enter/Space, Esc가 정의된 grid 패턴대로 동작한다.
**And** 오늘과 선택 날짜가 접근성 상태로 구분된다.

**Given** 사용자가 게시 날짜를 선택했을 때
**When** 해당 날짜의 작업이 존재하지 않으면
**Then** 내부 작업 ID를 가진 새 작업이 SQLite에 생성된다.
**And** 게시 날짜별 앱 관리 작업 폴더가 생성된다.

**Given** 작업 생성 또는 조회 요청이 renderer에서 전달될 때
**When** main과 Python 경계를 통과하면
**Then** 게시 날짜와 작업 ID가 각 경계에서 검증된다.
**And** renderer는 SQLite나 파일 시스템에 직접 접근하지 않는다.

**Given** 선택한 날짜에 저장된 작업이 있을 때
**When** 사용자가 앱을 다시 실행하거나 해당 날짜를 다시 선택하면
**Then** 저장된 작업 상태와 입력 메타데이터가 복원된다.
**And** 실제 파일 경로와 SQLite 기록이 대조된다.

**Given** 앱이 이전 실행 중 중단된 작업을 발견했을 때
**When** 시작 복구 절차가 실행되면
**Then** 작업은 `interrupted` 상태로 정리된다.
**And** 기존 입력과 완료 결과는 변경되지 않는다.

**Given** 날짜별 작업 저장 테스트를 실행할 때
**When** 빈 DB와 이전 스키마 DB에 마이그레이션을 적용하면
**Then** 마이그레이션이 순서대로 성공한다.
**And** 게시 날짜와 내부 작업 ID가 분리되어 저장된다.

### Story 1.4: 다중 입력 파일 등록과 안전한 보관

As a 개인 제작자,
I want 한 작업의 여러 입력 파일을 한 번에 등록하고 싶다,
So that 원본을 보호하면서 반복적인 파일 선택 시간을 줄일 수 있다.

**Requirements:** FR2, FR5; NFR7, NFR8; UX-DR6, UX-DR24

**Acceptance Criteria:**

**Given** 게시 날짜 작업이 선택되어 있을 때
**When** 사용자가 여러 파일을 Drop Zone에 놓으면
**Then** 모든 파일을 하나의 배치로 처리한다.
**And** 각 파일의 성공 또는 거부 결과를 독립적으로 표시한다.

**Given** 사용자가 드래그앤드롭을 사용하지 않을 때
**When** Drop Zone을 클릭하고 여러 파일을 선택하면
**Then** 드래그앤드롭과 동일한 등록 절차가 실행된다.
**And** 키보드만으로 파일 선택기를 열 수 있다.

**Given** 등록 가능한 파일이 전달되었을 때
**When** 파일 보관이 성공하면
**Then** 선택 게시 날짜의 `input` 폴더에 원본의 사본을 저장한다.
**And** 출처 파일은 수정하거나 삭제하지 않는다.

**Given** 지원하지 않거나 읽을 수 없는 파일이 포함되어 있을 때
**When** 배치 등록을 처리하면
**Then** 해당 파일만 거부한다.
**And** 다른 정상 파일은 계속 등록한다.

**Given** 앱 관리 폴더에 파일을 저장하려 할 때
**When** 대상 경로를 계산하면
**Then** 대상이 앱 관리 루트 내부인지 검증한다.
**And** 경로 문자열 조합 대신 플랫폼 경로 API를 사용한다.

**Given** 파일 배치 처리가 완료되었을 때
**When** 접근성 알림을 제공하면
**Then** 성공·거부 개수를 한 번만 알린다.
**And** 개별 오류는 해당 파일 상태와 연결한다.

**Given** 파일 복사 중 오류가 발생했을 때
**When** 등록 결과를 저장하면
**Then** 실패 파일을 입력 메타데이터에 정상 파일로 기록하지 않는다.
**And** 사용자가 다시 선택할 수 있는 행동 중심 오류를 표시한다.

### Story 1.5: 입력 파일 자동 분류와 슬롯 관리

As a 개인 제작자,
I want 등록한 파일이 역할별 슬롯에 자동 배치되고 문제 파일을 수정하고 싶다,
So that 각 파일의 용도를 직접 반복 지정하지 않고 입력을 준비할 수 있다.

**Requirements:** FR6, FR8; UX-DR7, UX-DR17

**Acceptance Criteria:**

**Given** 이미지 또는 텍스트 파일이 등록되었을 때
**When** 시스템이 파일 역할을 판별하면
**Then** 이미지 파일은 썸네일, 텍스트 파일은 스크립트 후보로 분류된다.
**And** 해당 File Slot에 역할과 파일명이 표시된다.

**Given** 오디오 파일이 등록되었을 때
**When** 파일명이 `voice.*` 또는 `bgm.*` 규칙과 일치하면
**Then** 음성 또는 작업별 BGM 슬롯에 각각 배치된다.
**And** 확장자만으로 불확실한 역할을 임의로 결정하지 않는다.

**Given** 파일 역할을 하나로 결정할 수 없을 때
**When** 분류 결과를 표시하면
**Then** 파일은 미분류 상태로 유지된다.
**And** 역할 선택 또는 제거 동작을 제공한다.

**Given** 동일 슬롯에 후보가 여러 개 있을 때
**When** 분류가 완료되면
**Then** 해당 슬롯을 충돌 상태로 표시한다.
**And** 충돌이 해결되기 전에는 임의 파일을 활성 입력으로 사용하지 않는다.

**Given** 사용자가 슬롯의 파일을 제거할 때
**When** 제거를 확인하면
**Then** SQLite 입력 메타데이터와 앱 관리 사본이 일관되게 제거된다.
**And** 삭제 대상이 앱 관리 루트 내부인지 검증한다.

**Given** 사용자가 슬롯 파일을 교체할 때
**When** 새 파일의 복사와 검증이 성공하면
**Then** 기존 사본과 메타데이터를 새 파일로 교체한다.
**And** 새 파일 처리 실패 시 기존 정상 입력을 유지한다.

**Given** File Slot이 정상·충돌·누락·미분류 상태일 때
**When** 화면에 상태를 표시하면
**Then** 색상과 함께 아이콘 및 행동 중심 문구를 제공한다.
**And** 각 제거·교체·역할 선택 동작은 키보드로 실행할 수 있다.

### Story 1.6: 스크립트와 생성 준비 상태 검증

**Given** 필수 입력에 누락·충돌·미분류·유효성 오류가 있을 때
**When** 사용자가 홈 화면을 확인하면
**Then** 생성 버튼은 비활성화된다.

**Given** 네 가지 입력 조건과 필요한 공통 리소스가 모두 유효할 때
**When** 준비 상태 계산이 완료되면
**Then** `준비 완료`, `100% · 필수 입력 4/4`를 표시한다.
**And** 영상 생성 버튼을 활성화한다.

**Given** 준비 상태가 갱신될 때
**When** 접근성 기술이 상태를 읽으면
**Then** 변경된 충족 상태를 과도하게 반복 낭독하지 않는다.
**And** 색상만으로 준비 여부를 전달하지 않는다.

**Given** 정상 입력 파일이 준비되어 있을 때
**When** 사용자가 게시 날짜 선택부터 파일 등록과 생성 버튼 활성화까지의 흐름을 수행하면
**Then** 사용자 조작 시간 5분 이내에 완료할 수 있는지 사용성 테스트로 측정한다.
**And** 기준을 넘으면 불필요한 단계와 반복 입력을 결함으로 기록한다.

### Story 1.7: 공통 리소스 설정과 작업별 BGM 적용

As a 개인 제작자,
I want 반복 사용하는 영상·음악·폰트를 설정하고 작업별 BGM을 선택적으로 적용하고 싶다,

**Given** 작업별 BGM이 등록되었을 때
**When** 현재 작업의 BGM 소스를 계산하면
**Then** 해당 작업에는 작업별 BGM을 우선 사용한다.
**And** 저장된 기본 BGM 설정은 변경하지 않는다.

**Given** 공통 리소스가 교체되었을 때
**When** 기존 완료 작업을 조회하면
**Then** 기존 결과 파일에는 변경이 소급 적용되지 않는다.
**And** 실행 중인 작업 스냅샷도 변경되지 않는다.

**Given** 사용자가 키보드와 스크린리더를 사용할 때
**When** 설정 모달을 열고 조작한 뒤 닫으면
**Then** 포커스가 모달 내부에 유지되고 닫힌 후 호출 버튼으로 돌아간다.
**And** 각 리소스 상태와 교체 버튼에 명확한 접근성 이름이 제공된다.

### Story 1.8: 작업 흐름을 설명하는 사용 가이드

As a 개인 제작자,

**When** 예시를 표시하면
**Then** `voice.*`, `bgm.*` 분류 규칙과 `[제목]`, `[말씀]`, `[기도]` 필수 구역을 설명한다.

## Epic 2: 게시 가능한 쇼츠 자동 생성

사용자는 준비된 작업으로 스크립트·음성·자막·영상·오디오 처리 과정을 실행하고 진행 상태를 확인한 뒤 게시 가능한 최종 MP4와 산출물을 받을 수 있다.

### Story 2.1: 검증된 작업 실행과 실시간 진행 상태

As a 개인 제작자,
I want 준비 완료된 작업의 생성을 시작하고 현재 단계를 확인하고 싶다,
So that 긴 처리 중에도 앱이 정상 동작하는지 알 수 있다.

**Requirements:** FR17, FR18; NFR2, NFR16, NFR17; UX-DR13, UX-DR18, UX-DR23

**Acceptance Criteria:**

**Given** 작업이 `준비 완료`이고 실행 중인 작업이 없을 때
**When** 사용자가 `영상 생성`을 선택하면
**Then** 입력과 공통 리소스의 실행 스냅샷을 만들고 Python 엔진에 `start` 명령을 전달한다.
**And** 명령은 `protocolVersion`, `type`, `jobId`, `timestamp`, `payload`를 포함한 JSON Lines 계약을 충족한다.

**Given** 작업 생성이 시작되었을 때
**When** 엔진이 단계 이벤트를 전송하면
**Then** 이벤트 어댑터가 이를 허용된 UI 상태 전이로 변환한다.
**And** `validating`, `analyzing`, `subtitling`, `composing`, `saving` 상태와 현재 단계명을 표시한다.

**Given** 진행 이벤트가 수신될 때
**When** 현재 실행 작업의 `jobId`와 일치하면
**Then** 전체 진행률을 단조 증가하도록 갱신한다.
**And** 완료 이벤트 전에는 100%를 표시하지 않는다.

**Given** 다른 작업 ID 또는 허용되지 않은 상태 전이의 이벤트가 수신될 때
**When** 이벤트 어댑터가 처리하면
**Then** 현재 UI 실행 상태에 적용하지 않는다.
**And** 진단 가능한 계약 위반으로 기록한다.

**Given** 작업이 실행 중일 때
**When** 사용자가 다른 작업을 시작하거나 현재 입력을 편집하려 하면
**Then** 새로운 생성 실행을 차단한다.
**And** 실행 스냅샷은 다음 작업용 편집 상태와 분리해 유지한다.

**Given** 생성 단계가 변경되었을 때
**When** UI가 이벤트를 수신하면
**Then** 1초 이내에 단계명과 진행률을 갱신한다.
**And** 단계 변경 또는 10% 단위 통과 시에만 polite live region으로 알린다.

### Story 2.2: 음성 기반 타이밍 생성

As a 개인 제작자,
I want 스크립트와 음성을 로컬에서 분석해 영상 구간과 자막 타이밍을 만들고 싶다,
So that 수동으로 자막 시점을 맞추지 않아도 된다.

**Requirements:** FR11, FR12; NFR3, NFR11, NFR12, NFR20

**Acceptance Criteria:**

**Given** 유효한 스크립트와 음성 파일이 있는 실행 작업일 때
**When** 엔진이 분석 단계를 시작하면
**Then** 스크립트의 `[제목]`, `[말씀]`, `[기도]` 내용과 기도 구역의 빈 줄·줄바꿈을 구조화한다.
**And** 파싱 결과 또는 오류를 작업 로그에 기록한다.

**Given** 음성 분석을 실행할 때
**When** 로컬 faster-whisper 런타임을 호출하면
**Then** 입력 파일을 외부 서버로 전송하지 않는다.
**And** 패키징된 로컬 모델만 사용해 네트워크 없이 처리한다.

**Given** 녹음 앞부분에 무음이 있을 때
**When** 타이밍을 계산하면
**Then** 앞부분 무음을 정리하고 영상 시작 후 약 2초에 음성이 시작되도록 기준을 만든다.

**Given** 분석된 음성과 스크립트가 정렬될 때

**And** 다음 블록과 잘못 겹치지 않도록 타이밍 순서를 검증한다.

**Given** 기도 첫 구절의 후보가 하나로 결정될 때

**And** `artifact_created` 이벤트와 분석 처리 시간을 기록한다.

**Given** 기도 첫 구절을 찾지 못하거나 후보가 여러 개일 때
**When** 기도 시작점을 하나로 결정할 수 없으면
**Then** 렌더링 파이프라인을 자막 생성 전에 중단한다.
**And** `PRAYER_BOUNDARY_AMBIGUOUS` 오류와 확인해야 할 스크립트·음성 구간을 기록한다.

**Given** 기준 샘플에서 음성 모델 설정을 평가할 때
**When** 모델 크기, 양자화 및 CPU thread 조합을 실행하면
**Then** 한국어 타이밍 정확도와 처리 시간을 기록한다.
**And** 선택된 설정을 타입이 지정된 런타임 설정으로 고정한다.

### Story 2.3: 스타일 자막 생성

As a 개인 제작자,

**Given** 패키징된 앱을 깨끗한 대상 환경에 설치했을 때
**When** 설치 스모크 테스트를 실행하면
**Then** 앱 시작, 사용자 데이터 초기화, SQLite 마이그레이션, 포함 바이너리 실행 및 샘플 렌더링이 성공한다.
**And** 사용자에게 Python, FFmpeg 또는 별도 Python 패키지 설치를 요구하지 않는다.

**Given** 네트워크 연결을 차단한 설치 환경일 때
**When** 입력 등록부터 샘플 영상 생성까지 실행하면
**Then** 외부 서비스 없이 전체 흐름이 완료된다.
**And** 입력 및 생성 데이터의 외부 전송이 발생하지 않는다.

**Given** 실제 사용 안정성을 검증할 때
**When** 동일 설치본으로 7일 연속 하루 한 개 이상의 영상을 생성하면
**Then** 치명적 데이터 손상 없이 각 작업을 완료하거나 복구 가능한 오류로 종료한다.
**And** 등록 입력, 기존 완료 결과 및 시도별 로그의 보존 상태를 확인한다.

**Given** 외부 사용자에게 설치 파일을 배포할 때
**When** 릴리스 게이트를 실행하면
**Then** 플랫폼 코드 서명과 macOS notarization을 확인한다.
**And** 자동 업데이트 기능은 MVP 설치본에 포함하지 않는다.

## Epic 3: 실패와 재생성에서 결과 보호

사용자는 실행 작업을 안전하게 취소하고 실패 원인과 해결 방법을 확인하며, 완료된 날짜의 기존 결과를 보존한 상태로 명시적인 재생성을 수행할 수 있다.

### Story 3.1: 실행 중 작업의 안전한 취소

As a 개인 제작자,
I want 잘못 시작한 생성 작업을 안전하게 취소하고 싶다,
So that 시스템 자원과 임시 파일을 남기지 않고 다시 준비할 수 있다.

**Requirements:** FR19; NFR4, NFR6, NFR9; UX-DR16, UX-DR18, UX-DR22

**Acceptance Criteria:**

**Given** 작업이 생성 중일 때
**When** 사용자가 취소를 선택하면
**Then** 취소 영향과 임시 파일 정리를 설명하는 확인 모달을 표시한다.
**And** 확인 전에는 실행 프로세스를 중단하지 않는다.

**Given** 사용자가 취소를 확인했을 때
**When** 취소 절차가 시작되면
**Then** 작업 상태를 `cancelling`으로 전환하고 새 작업 시작을 차단한다.
**And** JSON Lines `cancel` 명령을 현재 `jobId`에 전달한다.

**Given** 엔진이 정상 취소에 응답할 때
**When** 실행 중인 분석 또는 FFmpeg 처리를 종료하면
**Then** 단 하나의 `cancelled` 최종 이벤트를 전송한다.
**And** 취소된 작업을 완료 항목으로 기록하지 않는다.

**Given** 정상 취소 제한 시간 안에 엔진이 응답하지 않을 때
**When** 강제 종료 절차를 실행하면
**Then** Python, FFmpeg 및 음성 분석 하위 프로세스를 포함한 프로세스 트리를 종료한다.
**And** 종료 결과와 진단 정보를 시도 로그에 남긴다.

**Given** 작업이 취소되었을 때
**When** 정리 절차가 완료되면
**Then** 해당 시도의 임시 파일을 삭제하고 취소 로그를 보존한다.
**And** 기존 완료 결과와 등록 입력은 변경하지 않는다.

**Given** 취소 확인 모달을 키보드로 조작할 때
**When** Esc 또는 취소 버튼으로 닫으면
**Then** 실행 작업은 계속된다.
**And** 포커스가 취소를 호출한 버튼으로 돌아간다.

### Story 3.2: 생성 실패 진단과 복구 안내

As a 개인 제작자로서,
I want 생성 실패의 단계와 해결 방법을 이해하고 싶다,
So that 문제를 수정한 뒤 직접 다시 실행할 수 있다.

**Requirements:** FR12, FR21; NFR4, NFR6, NFR10, NFR14; UX-DR16, UX-DR18

**Acceptance Criteria:**

**Given** 입력, 분석, 자막, 합성, 저장 단계에서 오류가 발생했을 때
**When** 오류를 처리하면
**Then** 안정적인 오류 코드와 실패 단계를 기록한다.
**And** `completed` 또는 `cancelled`와 중복되지 않는 단 하나의 `failed` 최종 이벤트를 전송한다.

**Given** 기도 시작점을 찾지 못하거나 후보가 여러 개일 때
**When** 음성 분석을 중단하면
**Then** `PRAYER_BOUNDARY_AMBIGUOUS`로 분류한다.
**And** 스크립트와 음성의 첫 기도 문장을 확인하라는 안내를 제공한다.

**Given** main 프로세스가 Python 또는 subprocess 오류를 받았을 때
**When** renderer용 오류로 변환하면
**Then** `code`, `message`, `recoverable`, `details`를 가진 `AppError`로 매핑한다.
**And** 스택 트레이스와 원시 subprocess 오류를 UI에 직접 노출하지 않는다.

**Given** 실패 모달을 표시할 때
**When** 사용자가 오류 내용을 확인하면
**Then** 실패 단계, 이해 가능한 원인 및 입력 수정·설정 이동·로그 폴더 열기 중 관련 동작을 제공한다.
**And** 자동 재시도를 시작하지 않는다.

**Given** 생성이 실패했을 때
**When** 실패 정리를 수행하면
**Then** 임시 파일을 삭제하고 `logs/<attemptId>-render_log.txt`를 보존한다.
**And** 실패 작업을 완료 목록에 추가하지 않는다.

**Given** 엔진 프로세스가 비정상 종료되었을 때
**When** 앱이 이를 감지하거나 다시 시작되면
**Then** 현재 작업을 `interrupted`로 표시하고 처음부터 다시 실행하도록 안내한다.
**And** 실행 중 작업을 자동 재시도하거나 기존 결과를 변경하지 않는다.

### Story 3.3: 완료 날짜 보호와 원자적 재생성

As a 개인 제작자,
I want 완료된 날짜의 기존 영상을 보호하면서 명시적으로 다시 생성하고 싶다,
So that 수정본 생성이 실패해도 게시 가능한 기존 결과를 잃지 않는다.

**Requirements:** FR3, FR4; NFR6; UX-DR19

**Acceptance Criteria:**

**Given** 완료 기록이 있는 게시 날짜를 선택했을 때
**When** 홈 화면이 작업을 로드하면
**Then** 이미 생성된 날짜임을 표시하고 일반 생성 버튼을 비활성화한다.
**And** 기존 완료 정보, 결과 폴더 열기 및 `다시 생성` 동작을 제공한다.

**Given** 사용자가 `다시 생성`을 선택했을 때
**When** 확인 모달이 표시되면
**Then** 기존 결과가 성공 후 교체된다는 조건을 설명한다.
**And** 확인 전에는 기존 파일이나 완료 기록을 변경하지 않는다.

**Given** 사용자가 재생성을 확인했을 때
**When** 새 시도를 시작하면
**Then** 날짜별 보관 입력을 그대로 사용하거나 교체된 입력으로 새 실행 스냅샷을 만든다.
**And** 새 결과는 기존 `output`이 아닌 별도 임시 위치에 생성한다.

**Given** 재생성의 영상과 산출물 검증이 모두 성공했을 때
**When** 교체를 커밋하면
**Then** SQLite 트랜잭션과 원자적 파일 교체로 기존 결과를 새 결과로 변경한다.
**And** 완료 기록의 실제 생성일과 성공 로그를 갱신한다.

**Given** 재생성 중 실패·취소·비정상 종료 또는 결과 커밋 실패가 발생했을 때
**When** 시도를 종료하면
**Then** 기존 완료 영상, 완료 기록 및 성공 로그를 그대로 유지한다.
**And** 실패 시도 로그를 별도 `attemptId` 파일로 보존한다.

**Given** 게시 날짜 중복 검사를 실행할 때
**When** 일반 생성 경로로 같은 완료 날짜를 다시 생성하려 하면
**Then** DB 유일 제약과 애플리케이션 검증 모두에서 차단한다.
**And** 명시적인 재생성 경로만 허용한다.

## Epic 4: 완료 이력과 결과 관리

사용자는 최근 작업과 완료 이력을 확인하고 결과 폴더의 실제 상태를 파악하며 운영체제 파일 탐색기로 결과를 열 수 있다.

### Story 4.1: 최근 작업과 완료 목록 조회

As a 개인 제작자,
I want 최근 작업과 완료된 게시 날짜를 한눈에 확인하고 싶다,
So that 어디까지 제작했는지 빠르게 파악할 수 있다.

**Requirements:** FR22, FR23; NFR10; UX-DR12, UX-DR20, UX-DR29

**Acceptance Criteria:**

**Given** 완료 작업이 하나 이상 있을 때
**When** 앱을 실행하면
**Then** SQLite에서 완료 목록을 불러와 게시 날짜 내림차순으로 표시한다.
**And** 가장 최근 완료 항목을 기본 선택해 제목, 게시 날짜, 실제 생성일 및 결과 위치를 표시한다.

**Given** 마지막 작업이 실패·취소·중단 상태이고 이후 완료 작업이 없을 때
**When** 앱을 실행하면
**Then** 최근 작업 영역에 해당 작업의 제목, 게시 날짜, 상태 및 가능한 결과·로그 위치를 표시한다.
**And** 완료되지 않은 작업을 완료 목록 항목으로 표시하지 않는다.

**Given** 완료 이력이 없을 때
**When** 홈 화면을 표시하면
**Then** 새 작업 등록을 안내하는 빈 상태를 표시한다.
**And** 완료 목록 영역에는 첫 작업 안내를 제공한다.

**Given** 완료 항목을 표시할 때
**When** 제목이 행 너비보다 길면
**Then** 한 줄 말줄임표로 표시한다.
**And** 전체 제목은 툴팁 또는 접근성 이름으로 제공한다.

**Given** 사용자가 완료 행을 선택했을 때
**When** 선택 상태가 변경되면
**Then** 최근 작업 영역을 해당 항목 정보로 갱신한다.
**And** 행 선택만으로 결과 폴더를 열지 않는다.

**Given** 앱 시작 또는 목록 새로고침 시 결과 경로를 대조할 때
**When** 기록된 결과 폴더가 존재하지 않으면
**Then** `결과 폴더를 찾을 수 없습니다` 상태를 표시한다.
**And** 해당 항목을 정상 완료처럼 열 수 있는 상태로 표시하지 않는다.

**Given** 목록과 최근 작업 UI를 키보드로 탐색할 때
**When** Tab과 Enter/Space를 사용하면
**Then** 행 선택과 폴더 열기 버튼이 각각 별도 포커스 대상으로 동작한다.
**And** 포커스와 선택 상태를 색상 외의 방식으로 구분한다.

### Story 4.2: 운영체제 탐색기로 결과 폴더 열기

As a 개인 제작자,
I want 완료 결과 폴더를 운영체제 파일 탐색기로 열고 싶다,
So that 최종 영상을 재생·이동하거나 외부 작업에 사용할 수 있다.

**Requirements:** FR24; NFR15; UX-DR12, UX-DR20, UX-DR26

**Acceptance Criteria:**

**Given** 최근 작업 또는 완료 항목의 결과 폴더가 존재할 때
**When** 사용자가 `결과 폴더 열기`를 선택하면
**Then** Windows에서는 Explorer, macOS에서는 Finder로 해당 폴더를 연다.
**And** 한 번의 명시적 동작으로 수행한다.

**Given** renderer가 폴더 열기를 요청할 때
**When** preload와 main 경계를 통과하면
**Then** 작업 ID와 저장된 결과 경로를 다시 검증한다.
**And** renderer가 임의 경로를 직접 열거나 운영체제 API를 직접 호출하지 않는다.

**Given** 저장된 경로가 앱 관리 루트 밖을 가리키거나 존재하지 않을 때
**When** 폴더 열기를 요청하면
**Then** 운영체제 호출을 실행하지 않는다.
**And** 사용자가 취할 수 있는 오류 메시지를 표시한다.

**Given** Windows 또는 macOS에서 경로를 처리할 때
**When** 폴더 열기 명령을 구성하면
**Then** 플랫폼 경로 API와 Electron의 안전한 OS 통합 기능을 사용한다.
**And** 경로 구분자나 드라이브 문자를 UI 문구와 공통 코드에 하드코딩하지 않는다.

**Given** 결과 폴더를 연 뒤
**When** 사용자가 앱으로 돌아오면
**Then** 현재 선택 항목과 입력 편집 상태를 보존한다.
**And** 앱 내부 영상 재생·복제 다운로드·파일 삭제 기능은 제공하지 않는다.
