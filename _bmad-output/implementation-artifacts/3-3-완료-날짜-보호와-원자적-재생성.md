# Story 3.3: 완료 날짜 보호와 원자적 재생성

Status: done

## Story

As a 개인 제작자,
I want 완료된 날짜의 기존 영상을 보호하면서 명시적으로 다시 생성하고 싶다,
so that 수정본 생성이 실패해도 게시 가능한 기존 결과를 잃지 않는다.

## Acceptance Criteria

1. 완료 날짜를 로드하면 이미 생성됨을 표시하고 일반 생성은 막으며 완료 정보·폴더 열기·`다시 생성`을 제공한다.
2. 다시 생성 확인 모달은 성공 후 교체 조건을 설명하고 확인 전 기존 파일·기록을 변경하지 않는다.
3. 확인 후 보관/교체 입력으로 새 실행 스냅샷을 만들고 새 결과는 기존 output과 분리된 temp에 생성한다.
4. 새 영상·산출물 검증 성공 후 SQLite transaction과 atomic file replacement로 교체하고 실제 생성일·성공 로그를 갱신한다.
5. 실패·취소·crash·commit 실패 시 기존 완료 영상·기록·성공 로그를 유지하고 attemptId 실패 로그를 보존한다.
6. 일반 중복 생성은 DB unique와 앱 검증 양쪽에서 막고 명시 재생성 경로만 허용한다.

- [x] 모든 실패 지점의 rollback/reconciliation이 기존 output hash·DB·로그를 보존하게 한다. (AC: 5)
- [x] 중복 요청·동시 요청·disk failure·rename/DB crash fault injection 테스트를 추가한다. (AC: 1~6)

## Dev Notes

- 기존 결과 삭제가 먼저 일어나면 안 된다. 새 결과 검증, pending staging, 교체 가능성 확인 후 commit한다.
- 파일 교체와 SQLite transaction 사이 crash window를 startup reconciliation으로 복구할 수 있게 marker/상태를 설계한다. 새 manifest를 제품 계약으로 추가하지 않는다.
- DB의 publish date uniqueness를 우회하지 않는다. 재생성은 같은 job의 새 attempt로 모델링한다.
- 재생성 실패 로그는 기존 성공 `output/render_log.txt`를 덮지 않는다.
