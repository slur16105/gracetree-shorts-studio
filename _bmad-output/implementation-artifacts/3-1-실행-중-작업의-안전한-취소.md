# Story 3.1: 실행 중 작업의 안전한 취소

Status: review

## Story

As a 개인 제작자,
I want 잘못 시작한 생성 작업을 안전하게 취소하고 싶다,
so that 시스템 자원과 임시 파일을 남기지 않고 다시 준비할 수 있다.

## Acceptance Criteria

1. 생성 중 취소 선택 시 영향·temp 정리를 설명하는 확인 모달을 표시하고 확인 전에는 프로세스를 중단하지 않는다.
2. 확인 후 상태를 `cancelling`으로 바꾸고 새 시작을 막으며 현재 jobId에 JSON Lines `cancel`을 보낸다.
3. 정상 취소 시 분석·FFmpeg를 종료하고 단 하나의 `cancelled`를 보내며 완료로 기록하지 않는다.
4. 제한 시간 내 응답이 없으면 Python·FFmpeg·음성 분석 프로세스 트리를 종료하고 결과·진단을 attempt log에 남긴다.
5. 취소 temp를 삭제하고 취소 로그는 보존하며 기존 완료 결과와 입력은 변경하지 않는다.
6. 확인 모달을 Esc/취소 버튼으로 닫으면 실행을 계속하고 호출 버튼으로 포커스를 복원한다.

## Tasks / Subtasks

- [x] cancel command/event schema와 `running → cancelling → cancelled` 상태 전이를 추가한다. (AC: 2, 3)
- [x] Result Dialog 패턴을 재사용한 취소 확인과 포커스 동작을 구현한다. (AC: 1, 6)
- [x] main job-service가 현재 attempt에만 cancel을 보내고 grace timeout을 관리하게 한다. (AC: 2~4)
- [x] Python orchestrator가 cancellation token을 각 단계와 child process에 전달하고 terminal event를 단일화한다. (AC: 3, 4)
- [x] 플랫폼별 process-tree 종료 어댑터와 진단·temp 정리를 구현한다. (AC: 4, 5)
- [x] cancel/completed race, timeout, repeated cancel, app close 중 취소 테스트를 추가한다. (AC: 1~6)

## Dev Notes

- Esc는 모달만 닫고 실행 자체를 취소하지 않는다.
- 정상 취소를 먼저 시도한 뒤 제한 시간 경과 시에만 강제 종료한다. PID 재사용·다른 작업 프로세스 오종료를 막기 위해 현재 attempt가 생성한 process handle/tree만 관리한다.
- cancelled 이후 늦게 도착한 completed/progress는 상태 머신에서 거부·기록한다.
- 기존 output·input·성공 로그는 삭제하지 않는다. 정리 범위는 현재 attempt temp로 제한한다.
