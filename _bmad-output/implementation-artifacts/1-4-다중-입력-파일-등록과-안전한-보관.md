# Story 1.4: 다중 입력 파일 등록과 안전한 보관

Status: ready-for-dev

## Story

As a 개인 제작자,
I want 한 작업의 여러 입력 파일을 한 번에 등록하고 싶다,
so that 원본을 보호하면서 반복적인 파일 선택 시간을 줄일 수 있다.

## Acceptance Criteria

1. Drop Zone의 여러 파일을 한 배치로 처리하고 파일별 성공·거부 결과를 독립 표시한다.
2. 클릭·키보드 파일 선택은 드래그앤드롭과 동일한 등록 절차를 사용한다.
3. 성공 파일은 선택 날짜 `input` 폴더에 사본으로 저장하고 출처 파일은 수정·삭제하지 않는다.
4. 지원하지 않거나 읽을 수 없는 파일만 거부하고 나머지 정상 파일은 계속 등록한다.
5. 대상이 앱 관리 루트 내부인지 검증하고 플랫폼 경로 API를 사용한다.
6. 배치 완료 시 성공·거부 개수를 한 번 알리고 개별 오류를 해당 파일 상태와 연결한다.
7. 복사 실패 파일은 정상 메타데이터로 기록하지 않고 재선택 동작을 제공한다.
8. 순차 마이그레이션으로 `job_inputs(id, job_id, role, original_name, managed_path, status, created_at, updated_at)`를 만들고 FK cascade, `uq_job_inputs_job_id_managed_path`, `idx_job_inputs_job_id`를 적용한다.

## Dev Notes

- 승인된 Sprint Change Proposal이 이 Story에 `job_inputs` 스키마 최초 소유권을 부여한다.
- renderer에서 드롭된 경로를 신뢰하지 않는다. main과 Python에서 작업 ID, 원본 경로, 최종 관리 경로를 재검증한다.
- 배치 전체 트랜잭션으로 정상 파일까지 롤백하지 않는다. 각 파일은 독립 결과를 갖되 파일 커밋과 DB insert는 해당 파일 단위로 일관되어야 한다.
- 동일 이름 충돌을 묵시적으로 덮어쓰지 않는다. Story 1.5의 슬롯 교체 확인 경로로 넘길 수 있는 충돌 상태를 반환한다.
- 새 런타임 의존성이나 파일 sniffing 라이브러리가 필요하면 먼저 승인받는다.
