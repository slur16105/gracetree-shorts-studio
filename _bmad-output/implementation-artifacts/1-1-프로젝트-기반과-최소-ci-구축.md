# Story 1.1: 프로젝트 기반과 최소 CI 구축

Status: done

## Story

As a 개발 담당자,
I want 승인된 프로젝트 구조와 자동 검증 기반을 구축하고 싶다,
so that 이후 기능을 일관된 경계에서 안전하게 구현할 수 있다.

## Acceptance Criteria

1. **Given** Node.js와 pnpm 개발 환경이 준비되어 있을 때
   **When** Electron React/TypeScript 스타터를 초기화하면
   **Then** `apps/desktop` 애플리케이션을 개발 모드로 실행할 수 있다.
   **And** Electron main, preload, renderer 구조가 분리된다.

2. **Given** 프로젝트 초기화가 완료되었을 때
   **When** 저장소 구조를 확인하면
   **Then** pnpm 워크스페이스에 `apps/desktop`과 `packages/contracts`가 포함된다.
   **And** 별도 Python 엔진 패키지와 테스트 디렉터리가 존재한다.

3. **Given** Electron과 Python 엔진이 통신할 때
   **When** 상태 확인용 JSON Lines 메시지를 교환하면
   **Then** 메시지는 `protocolVersion`, `type`, `jobId`, `timestamp`, `payload`를 포함한다.
   **And** 양쪽 구현이 동일한 JSON Schema로 메시지를 검증한다.
   **And** 일반 진단 로그는 stdout 프로토콜과 분리된다.

4. **Given** 최소 CI 워크플로가 실행될 때
   **When** 변경 사항을 검증하면
   **Then** TypeScript 타입 검사와 단위 테스트가 실행된다.
   **And** Python 테스트와 양방향 프로토콜 계약 테스트가 실행된다.
   **And** 검증 실패 시 CI가 실패한다.

5. **Given** 의존성이 설치되었을 때
   **When** lockfile을 확인하면
   **Then** Node 의존성 버전이 고정된다.
   **And** Python 의존성도 재현 가능한 방식으로 고정된다.

## Dev Notes

### Developer Context

- 이 Story는 제품 기능 구현 전의 기반 Story다. 현재 저장소에는 제품 코드, Node workspace, Python package, 테스트 설정이 없으므로 기존 구현을 수정하지 않고 승인된 구조를 새로 만든다.
- Epic 1의 후속 Story는 이 Story의 경계를 전제로 한다. main/preload/renderer 분리, 공유 계약 단일 소유권, Python 엔진 독립 패키지, 재현 가능한 설치와 CI가 완료되지 않은 상태에서 기능 코드를 앞당겨 구현하지 않는다.
- 상태 확인 메시지는 전체 작업 프로토콜과 장기 실행 엔진 수명주기의 최소 증명이다. 프로세스는 EOF까지 여러 command를 처리하되 `start`, 진행률, SQLite, FFmpeg 흐름을 미리 구현하지 않는다.
- JSON Schema가 계약의 단일 기준이다. TypeScript 타입과 Python 모델을 서로 독립적으로 정의해 drift를 허용하지 않는다.
- 현재 로컬 관측값은 Node `v26.0.0`, npm `11.12.1`, Python `3.9.6`, pnpm 미설치다. 이는 구현 완료 상태가 아니다. CI 기준은 pnpm 11과 electron-vite 5를 함께 지원하는 Node 22 계열로 고정하고, Python 기준은 `tomllib`을 포함하는 3.11 계열로 고정한다.

### Technical Requirements

- Node 패키지 관리자는 pnpm 11.8 계열만 사용하고 실제 버전은 루트 `packageManager` 및 lockfile로 고정한다.
- Electron 코드는 TypeScript, renderer UI는 React를 사용한다. renderer에 Node.js, 파일 시스템, SQLite, 프로세스 직접 접근을 추가하지 않는다.
- 프로세스 메시지는 한 줄당 한 JSON 객체인 JSON Lines 형식이며 공통 envelope는 다음 필드를 가진다.

```json
{
  "protocolVersion": 1,
  "type": "check_health",
  "jobId": "health-check",
  "timestamp": "2026-06-19T07:00:00.000Z",
  "payload": {}
}
```

- JSON 필드는 `camelCase`, 이벤트 타입은 발생 사실을 나타내는 `snake_case`를 사용한다. 진단 로그는 stdout이 아닌 stderr에 기록한다.
- `timestamp`는 UTC ISO 8601을 사용한다. 누락 값을 빈 문자열이나 임의 기본값으로 정상화하지 말고 schema validation을 실패시킨다.
- 상태 확인 메시지는 command `check_health`, event `health_checked`를 사용하고 schema, 타입, fixture, 양쪽 테스트에서 동일하게 유지한다.
- TypeScript JSON Schema 검증에는 `ajv`, UTC ISO 8601 `date-time` 형식 검증에는 `ajv-formats`, Python 검증에는 `jsonschema`를 사용한다. 이 세 런타임 의존성은 Story 1.1 범위로 승인되었으며 정확한 버전은 각 lockfile로 고정한다.
- 임의의 부분 검증기를 직접 만들어 JSON Schema 요구사항을 우회하지 않는다.
- Python stdout에 배너, 로그, traceback 또는 테스트 디버그 출력을 섞지 않는다. 오류는 stderr와 실패 종료 코드로 구분한다.
- Python 엔진은 앱 시작 시 생성되는 장기 실행 단일 프로세스를 전제로 EOF까지 command를 처리한다. Story 1.1의 Node 통합 harness도 하나의 프로세스에 복수 `check_health` command를 보내 각각 하나의 `health_checked` event를 받는 경로를 검증한다.
- Node와 Python의 정확한 의존 버전은 lockfile로 고정한다. CI는 lockfile 불일치를 허용하지 않는다.

### Architecture Compliance

- 초기화 명령과 물리적 구조는 Architecture의 starter 및 프로젝트 구조 결정을 따른다.
- `packages/contracts`가 TypeScript/Python 경계의 schema와 fixture를 소유한다.
- `apps/desktop`은 Electron main/preload/renderer 경계를 유지한다. 이 Story에서는 좁은 preload API의 제품 기능을 아직 설계하지 않는다.
- `engine/gracetree_engine`은 Python 엔진의 유일한 제품 패키지다. 테스트는 `engine/tests`에서 소스 구조를 반영한다.
- 공유 유틸리티는 실제 중복이 생기기 전에 만들지 않는다.
- 데이터베이스 및 `001_create_jobs.sql`은 Story 1.3 소유이므로 이 Story에서 생성하지 않는다.
- electron-builder, PyInstaller, FFmpeg, faster-whisper 리소스는 후속 패키징·생성 Story 소유이므로 빈 placeholder나 가짜 바이너리를 추가하지 않는다.

### Library / Framework Requirements

- 스타터: `@quick-start/electron`의 `react-ts` 템플릿.
- 워크스페이스: pnpm 11.8 계열. Architecture의 승인 버전 범위를 우선하며 단순히 시스템의 최신 pnpm으로 올리지 않는다.
- TypeScript 테스트: Vitest 4 계열. renderer 컴포넌트 테스트는 후속 UI Story에서 React Testing Library를 추가한다.
- Python 테스트: pytest.
- Schema dialect는 프로젝트 전체에서 하나로 고정하고 schema의 `$schema`에 명시한다. JSON Schema 공식 문서의 현재 메타 스키마를 임의로 섞지 않는다.
- Schema dialect는 JSON Schema Draft 2020-12로 고정한다. TypeScript는 Ajv의 2020-12 전용 인스턴스를 사용하고 Python은 `Draft202012Validator`를 명시한다.
- Electron, electron-vite, Vite, React의 정확한 버전은 스타터 생성 결과와 lockfile로 고정한다. Architecture 변경 승인 없이 수동 업그레이드하지 않는다.

### Testing Requirements

- TypeScript:
  - 계약 schema 자체가 파싱되는지 검증한다.
  - 정상 fixture 수락, 필수 필드 누락, 잘못된 필드 타입, 지원하지 않는 메시지 타입 거부를 검증한다.
  - Electron 스타터의 최소 코드가 타입 검사와 단위 테스트를 통과하는지 확인한다.
- Python:
  - 동일 fixture에 대해 TypeScript와 같은 수락/거부 결과를 검증한다.
  - 장기 실행 프로세스에서 stdin의 유효한 한 줄 입력당 stdout 한 줄 응답을 검증한다.
  - 하나의 프로세스에 연속으로 두 개 이상의 상태 확인 command를 보내 순서대로 응답하는지 검증한다.
  - 진단 텍스트가 stderr에만 기록되는지 검증한다.
- 통합:
  - Electron 측 또는 Node harness가 Python module을 shell string이 아닌 실행 파일과 인자 배열로 호출해 상태 확인 왕복을 검증한다.
  - 비정상 JSON과 schema-invalid 메시지는 성공 응답으로 위장하지 않고 비정상 종료 또는 결정적인 오류 경로로 처리한다.
- CI:
  - clean checkout과 frozen lockfile을 기준으로 실행한다.
  - 캐시가 비어 있어도 성공해야 한다.
  - 테스트를 통과시키기 위해 타입 검사나 계약 테스트를 비활성화하지 않는다.

### Scope Boundaries

- 포함: 워크스페이스, Electron 스타터, 공유 schema/fixture, Python package skeleton, 최소 JSON Lines 상태 확인, 타입 검사·테스트·CI, lockfile.

### Latest Technical Notes

- electron-vite 공식 가이드는 `npm create @quick-start/electron@latest`를 공식 scaffold 경로로 안내한다. 이 저장소는 Architecture가 승인한 `react-ts` 명령을 그대로 사용한다.
- electron-vite 공식 가이드의 현재 Node 최소 요구사항을 구현 시 다시 확인하고 CI Node 버전을 그 요구사항과 호환되는 고정 버전으로 선택한다. 로컬 Node `v26.0.0`을 검증 없이 CI 기준으로 채택하지 않는다.
- pnpm 공식 호환성 표에서 pnpm 11은 지원되는 Node 계열과 함께 사용해야 한다. Architecture가 pnpm 11.8 계열을 고정하므로 Corepack 또는 명시적 설치로 같은 버전을 재현한다.
- pnpm 11은 Node 22 이상을 요구하고 electron-vite 5는 Node 22.12 이상을 지원하므로 CI Node 기준은 22.12 이상인 Node 22 계열로 고정한다.
- Ajv에서 Draft 2020-12는 기본 draft-07 인스턴스와 구분되므로 2020 전용 entry point를 사용한다. Python은 schema의 `$schema`에 의존한 암시적 선택 대신 `Draft202012Validator`를 사용한다.
- GitHub Actions의 Node 워크플로는 dependency cache를 lockfile에 연결하되 의존성 자체를 캐시하지 않는다. CI의 기준은 cache hit가 아니라 frozen lockfile 설치와 검증 명령의 종료 코드다.
