# Story 1.3: 게시 날짜별 작업 생성과 복원

Status: ready-for-dev

## Story

As a 개인 제작자,
I want 쇼츠의 게시 날짜를 선택하고 날짜별 작업을 보존하고 싶다,
so that 앱을 다시 실행해도 준비하던 작업을 이어갈 수 있다.

## Acceptance Criteria

1. 홈의 게시 날짜 기본값은 현재 날짜이며 `YYYY-MM-DD · Day` 형식으로 표시된다.
2. 달력은 화살표, Home/End, PageUp/PageDown, Enter/Space, Esc의 접근성 grid 패턴을 지원하고 오늘과 선택일을 구분한다.
3. 선택 날짜의 작업이 없으면 내부 작업 ID와 SQLite 레코드, 앱 관리 작업 폴더를 생성한다.
4. renderer 요청의 게시 날짜와 작업 ID를 main과 Python 경계에서 각각 검증하며 renderer는 SQLite나 파일 시스템에 직접 접근하지 않는다.
5. 앱 재실행 또는 날짜 재선택 시 저장 상태와 입력 메타데이터를 복원하고 실제 경로와 SQLite 기록을 대조한다.
6. 빈 DB와 이전 스키마 DB에 순차 마이그레이션을 적용하며 게시 날짜와 내부 작업 ID를 분리해 저장한다.

## Dev Notes

- SQLite의 최초 소유 Story다. 별도 Node SQLite 모듈을 추가하지 않고 Python 표준 `sqlite3`만 사용한다.
- 게시 날짜는 `YYYY-MM-DD`, 시각은 UTC ISO 8601이다. 폴더명과 DB ID를 동일시하지 않는다.
- SQLite가 메타데이터 기준이고 파일 시스템은 입력·결과 존재의 기준이다. 불일치는 숨기지 말고 DTO 상태로 투영한다.
- `app.getPath('userData')` 아래 승인된 앱 관리 루트를 main에서 결정하되 Python에도 정규화된 루트와 ID를 전달하고 재검증한다.
- 달력은 roving tabindex를 사용해 한 날짜 셀만 tab 순서에 둔다. APG 예제를 그대로 복사하지 말고 실제 보조기술 테스트를 수행한다.
- Story 1.4의 `job_inputs` 테이블과 파일 등록, Story 3.3의 완료 날짜 재생성은 선행 구현하지 않는다.
