# Story 1.5: 입력 파일 자동 분류와 슬롯 관리

Status: ready-for-dev

## Story

As a 개인 제작자,
I want 등록한 파일이 역할별 슬롯에 자동 배치되고 문제 파일을 수정하고 싶다,
so that 각 파일의 용도를 직접 반복 지정하지 않고 입력을 준비할 수 있다.

## Acceptance Criteria

1. 이미지와 텍스트를 썸네일·스크립트 후보로 분류하고 역할·파일명을 File Slot에 표시한다.
2. `voice.*`, `bgm.*` 파일명 규칙으로 오디오 역할을 배치하며 확장자만으로 불확실한 역할을 결정하지 않는다.
3. 역할이 불명확하면 미분류로 유지하고 역할 선택 또는 제거를 제공한다.
4. 동일 슬롯 후보가 여러 개면 충돌로 표시하고 임의 활성 입력을 선택하지 않는다.
5. 제거 시 SQLite 메타데이터와 관리 사본을 일관되게 제거하고 관리 루트를 검증한다.
6. 교체는 새 파일 복사·검증 성공 후 커밋하며 실패 시 기존 정상 입력을 유지한다.
7. 정상·충돌·누락·미분류 상태를 색상 외 아이콘·문구로 제공하고 모든 동작을 키보드로 실행할 수 있다.

## Tasks / Subtasks

- [ ] 순수·결정적 input classifier와 명시적 role enum을 공유 계약에 정의한다. (AC: 1~4)
- [ ] repository에서 후보·활성·충돌·미분류 상태를 보존하고 DTO로 투영한다. (AC: 3~5)
- [ ] 제거는 관리 루트 검증 후 DB와 파일을 일관되게 갱신하고 실패를 숨기지 않는다. (AC: 5)
- [ ] 교체는 새 사본을 별도 임시 경로에서 검증한 뒤 원자적으로 메타데이터를 전환한다. (AC: 6)
- [ ] File Slot UI, 역할 선택, 제거·교체 확인과 키보드 동작을 구현한다. (AC: 1~7)
- [ ] 분류 표, 충돌, 제거·교체 실패, 접근성 회귀 테스트를 추가한다. (AC: 1~7)

## Dev Notes

- 파일 확장자와 MIME은 후보 판단 자료이지 신뢰 경계가 아니다. 실제 파일 가독성 검증과 파일명 규칙을 함께 적용한다.
- 자동 분류가 확실하지 않으면 미분류가 정답이다. “첫 파일 우선” fallback을 만들지 않는다.
- 제거·교체는 데이터 삭제를 포함하므로 Story AC가 허용한 앱 관리 사본에만 제한한다. 출처 파일과 기존 정상 입력은 보호한다.
- 상태는 `ready`, `missing`, `conflict`, `unclassified`, `invalid`처럼 계약으로 고정하고 UI 문자열과 분리한다.
