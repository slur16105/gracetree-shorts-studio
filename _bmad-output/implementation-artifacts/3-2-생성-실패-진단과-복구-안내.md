# Story 3.2: 생성 실패 진단과 복구 안내

Status: done

## Story

As a 개인 제작자로서,
I want 생성 실패의 단계와 해결 방법을 이해하고 싶다,
so that 문제를 수정한 뒤 직접 다시 실행할 수 있다.

## Acceptance Criteria

1. 입력·분석·자막·합성·저장 오류에 안정 코드와 실패 단계를 기록하고 다른 terminal event와 중복되지 않는 단 하나의 failed를 보낸다.
2. 기도 경계 후보가 0/N이면 `PRAYER_BOUNDARY_AMBIGUOUS`와 스크립트·음성 첫 기도 문장 확인 안내를 제공한다.
3. main은 Python/subprocess 오류를 `code`, `message`, `recoverable`, `details`의 AppError로 매핑하고 stack/raw 오류를 UI에 노출하지 않는다.
4. 실패 모달에 단계·이해 가능한 원인과 입력 수정·설정 이동·로그 폴더 열기 중 관련 동작을 제공하며 자동 재시도하지 않는다.
5. 실패 temp를 삭제하고 `logs/<attemptId>-render_log.txt`를 보존하며 완료 목록에 추가하지 않는다.
6. 비정상 엔진 종료를 `interrupted`로 표시하고 처음부터 재실행을 안내하며 자동 재시도·기존 결과 변경을 금지하고 필요한 필드를 순차 마이그레이션으로 추가한다.

## Tasks / Subtasks

- [x] 오류 code/stage/recoverable/details 계약과 terminal event invariant를 확장한다. (AC: 1~3)
- [x] 필요한 interrupted/failure DB 열을 다음 순차 migration으로 추가한다. (AC: 1, 6)
- [x] Python diagnostics가 민감정보 없는 기술 원인과 stage를 attempt log에 기록하게 한다. (AC: 1, 2, 5)
- [x] main error mapper와 renderer action mapping을 구현한다. (AC: 3, 4)
- [x] Failed Result Dialog, 로그 폴더 열기, 입력/설정 이동과 수동 재실행 흐름을 구현한다. (AC: 4)
- [x] startup reconciliation에서 running/cancelling 잔여 상태를 interrupted로 변환한다. (AC: 6)
- [x] 단계별 fault injection, process crash, terminal race, log 보존 테스트를 추가한다. (AC: 1~6)

## Dev Notes

- UI는 행동 가능한 한국어 메시지를 표시하고 error code는 진단 식별자로 유지한다. raw stderr·stack·절대 경로를 노출하지 않는다.
- 자동 재시도는 MVP 비목표다. “다시 실행”은 사용자가 입력을 확인한 뒤 명시적으로 시작한다.
- 실패·중단 attempt는 완료 목록에 포함하지 않는다. 기존 완료 output은 어떤 실패 정리에서도 변경하지 않는다.
